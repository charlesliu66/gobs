## 三、经验教训

1. **状态源不统一时，UI 很容易出现“结果正确、状态错误”的错位。**
   5/6 的 `v0.137` 本质上不是视频生成失败，而是“任务状态”和“结果状态”在不同层级被分别消费。以后凡是“状态条 / 预览 / 历史版本”来自不同数据源，应该先定义优先级合同，再做 UI 表达。

2. **平台语义切换是系统级变更，不是单点替换。**
   从 Dreamina CLI 切到 Ark Seedance API 后，同一天连续出现并发、ETA、状态命名、文案、验证记录等多轮收口，说明这种切换会连带影响调度、UI、文档和 smoke 口径，应该预先按“状态合同 + 队列合同 + 用户语言合同”成组设计。

3. **Campaign 能力建设速度很快，但开始出现“底座、策略、执行链路”并行推进的复杂度。**
   这说明知识底座方向是对的，但后续如果没有更清晰的 phase 边界，容易出现 plan 很多、线很多、验证成本上升的问题。

4. **当天文档质量整体不错，但 PRODUCT 健康度需要盯。**
   `PRODUCT.md` 在 5/6 已推进到 `v0.140`，版本记录很活跃，但也意味着更要警惕重复版本、附录漂移和“commit 已落但 changelog 口径不一致”的风险。
