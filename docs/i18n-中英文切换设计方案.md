## 1. 总体架构

### 1.1 语言切换模型

```
┌────────────────────────────────────────────────────────┐
│  用户偏好（localStorage / 后端 user profile）           │
│  locale: "zh-CN" | "en"                                │
└─────────────────┬──────────────────────────────────────┘
                  │
    ┌─────────────┼─────────────────┐
    ▼             ▼                 ▼
 前端 UI      Prompt 生成       后端消息
 (i18next)   (双语 system      (error msg
              prompt 模板)      国际化)
```

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **UI 与内容分离** | 界面标签用 i18n key，但剧本/prompt **内容** 的语言由 `contentLocale` 单独控制 |
| **一个偏好，两个维度** | `uiLocale`（界面语言）+ `contentLocale`（AI 输出内容语言），默认跟随 `uiLocale`，可独立覆盖 |
| **渐进迁移** | 不做一次性全量替换；先核心流程、再周边页面 |
| **兼容不破坏** | 所有中文已有数据（剧本、历史记录、素材标签）保持不变，英文只对新内容生效 |

### 1.3 为什么要两个维度？

场景：一个 local 同事想用英文界面，但生成的视频面向中国市场，prompt 仍需中文。反过来：中文界面的用户可能要给 TikTok 海外号做英文 caption。所以：

- `uiLocale`：控制所有 UI 按钮、导航、Toast、错误提示
- `contentLocale`：控制 AI 产出的剧本文案、prompt 文案、TikTok caption 语言

当前代码有 `toLocaleString('zh-CN', …)` 硬编码（如 `GalleryView.tsx` 第 33 行）。需改为：

```typescript
const formatted = new Date(ts).toLocaleString(i18n.language, {
  year: 'numeric', month: '2-digit', day: '2-digit',
  hour: '2-digit', minute: '2-digit'

```

### 2.5 语言切换入口

推荐在 `Layout.tsx` 顶部导航栏右侧添加语言切换按钮（地球图标 + 下拉）：

```

```

切换后：
1. `localStorage.setItem('ui_locale', 'en')`
2. `i18n.changeLanguage('en')`
3. 如果用户已登录，同步到后端 user profile（`PATCH /api/auth/preferences`）
4. 页面不刷新，React re-render 即时生效

## 3. 剧本 & Prompt 双语体系

### 3.1 当前剧本结构（中文）

```
[00:00-00:03] 镜头1：孤岛黎明（远景，缓慢上推）。金色晨光照亮荒芜小岛，椰树摇曳。
[00:03-00:06] 镜头2：主角特写（特写，静止）。主角睁开眼，满脸泥土与刮伤。
```

### 3.2 英文模式下的剧本格式

```
[00:00-00:03] Shot 1: Island Dawn (wide, slow push up). Golden morning light on a barren island, swaying palms.
[00:03-00:06] Shot 2: Hero Close-up (close-up, static). Hero opens eyes, dirt and scratches on face.
```

### 3.3 解析器需要支持双语

`storyboardParser.ts` 当前正则：

```regex
/\[(\d{2}):(\d{2})-(\d{2}):(\d{2})\]\s*镜头\d+[：:]\s*(.+?)[。.]([\s\S]*?)(?=\[\d{2}:\d{2}-\d{2}:\d{2}\]|$)/g
```

**坑点**：正则里硬编码了 `镜头`（中文）和 `。`（中文句号）。英文版需要匹配 `Shot` 和 `.`。

**改造方案**：

```regex
/\[(\d{2}):(\d{2})-(\d{2}):(\d{2})\]\s*(?:镜头|Shot)\s*\d+[：:.]\s*(.+?)[。.]([\s\S]*?)(?=\[\d{2}:\d{2}-\d{2}:\d{2}\]|$)/g
```

- `镜头|Shot`：双语关键字
- `[：:.]`：中文冒号 + 英文冒号 + 英文句号
- `[。.]`：中文句号 + 英文句号
- fallback 名称：`镜头${n}` → `contentLocale === 'en' ? `Shot ${n}` : `镜头${n}``

### 3.4 制片系统（Studio Pipeline）字段

`ProductionShot` 的字段 `sceneRef`, `subject`, `action`, `dialogue` 等是纯文本。**无需结构性改动**，只需让 LLM 产出对应语言内容。关键改动点在 LLM system prompt：

| 文件 | system prompt 变量 | 改动 |
|------|-------------------|------|
| `studioPipeline.ts` | `STORY_ARC_SYSTEM` | "中文优先" → 根据 contentLocale 切换 |
| `studioPipeline.ts` | `PRODUCTION_DESIGN_SYSTEM` | 同上 |
| `studioPipeline.ts` | `STORYBOARD_TABLE_SYSTEM` | 同上 |
| `promptPolish.ts` | `VEO_SINGLE_SHOT_SYSTEM` | "默认中文" → 根据 contentLocale |
| `promptPolish.ts` | `VEO_MULTI_SHOT_SYSTEM` | 同上 |
| `promptPolish.ts` | `SHORT_DRAMA_EXPAND_SYSTEM` | "全部使用中文" → 条件化 |
| `dreaminaPromptComposer.ts` | 重写器 system prompt | "结构化中文 Prompt" → 双语 |
| `shotReview.ts` | `SHOT_REVIEW_SYSTEM` | 中文导演指令 → 双语 |
| `shotReview.ts` | `CONTINUITY_SYSTEM` | 同上 |
| `editorAgentChat.ts` | system prompt | "用中文回答" → 根据 locale |
| `riskSentimentService.ts` | `ANALYSIS_SYSTEM` | 中文 JSON schema 描述 → 双语 |

**实现模式**：为 system prompt 创建工厂函数：

```typescript
function getStoryArcSystem(locale: ContentLocale): string {
  if (locale === 'en') {
    return `You are a narrative architect. Output a story arc in English...`;
  }
  return `你是叙事架构师。输出中文故事弧线...`; // 现有 prompt
}
```

### 3.5 ⚠️ 坑点：LLM 对中英文 prompt 质量差异

| 问题 | 详情 | 建议 |
|------|------|------|
| **Gemini 中英文输出质量** | Gemini 系列对英文的理解和遵循指令能力通常更强，但在生成游戏/影视中文描述时有独特优势 | 英文模式下可能需要调优 temperature 和示例 |
| **token 计算差异** | 中文 1 个字 ≈ 2-3 tokens，英文 1 个词 ≈ 1-1.5 tokens。同样的场景描述，英文 token 数可能只有中文的 60% | `maxTokens` 设置可能需要按语言微调 |
| **VEO prompt 语言** | VEO 对英文 prompt 的响应质量**通常更好**（原生英文模型） | 即使 `contentLocale=zh`，VEO 最终 prompt 也可考虑翻译为英文提交 |

## 4. 多模态引用 `@图片N` 的英文化

### 4.1 现状梳理

代码库中存在 **三套引用惯例**：

| 引用风格 | 使用场景 | 文件位置 |
|----------|---------|---------|
| `@图片N` / `@视频N` / `@音频N` | 即梦全能参考 / Tab 生成 / Dreamina 多模态 | `DreaminaMultimodalRefs.tsx`, `dreaminaPromptComposer.ts`, `productionAssets.ts` |
| `@img1` / `@img2` | 制片 multishot assemblePrompts | `studioPipeline.ts` |
| `【图片1】` / `【TikTok视频】` | 可灵 Omni Viral 舞蹈 | `promptPolish.ts` |

### 4.2 设计方案

**核心决策**：统一使用 **`@imageN` / `@videoN` / `@audioN`** 作为英文模式的引用标识，保持中文模式不变。

```typescript
// 引用标签工厂
function getRefTag(kind: 'image' | 'video' | 'audio', index: number, locale: ContentLocale): string {
  const labels = {
    zh: { image: '图片', video: '视频', audio: '音频' },
    en: { image: 'image', video: 'video', audio: 'audio' },
  };
  return `@${labels[locale][kind]}${index}`;
}
```

### 4.3 各模块改动清单

#### A. `DreaminaMultimodalRefs.tsx`（前端）

当前：
```typescript
if (kind === 'image') return { tag: `图${indexAmongKind + 1}`, hint: '@图片' + (indexAmongKind + 1) };
```

改为：
```typescript
if (kind === 'image') {
  const label = locale === 'en' ? 'image' : '图片';
  return { tag: locale === 'en' ? `img${idx}` : `图${idx}`, hint: `@${label}${idx}` };
}
```

#### B. `dreaminaPromptComposer.ts`（后端）

当前结构化检测：
```typescript
const hasAtRef = /@图片\d|@视频\d|@音频\d/.test(text);
```

改为双语正则：
```typescript
const hasAtRef = /@(?:图片|image)\d|@(?:视频|video)\d|@(?:音频|audio)\d/i.test(text);
```

重写器 system prompt 也要支持两套引用格式。

#### C. `productionAssets.ts`（制片系统）

`injectAtImageTagsInNarrative` 中的 `@图片${k}` → 根据 locale 选择 `@image${k}` 或 `@图片${k}`。

`buildShotMultimodalRefPack` 中的映射说明行：
```
参考对应：@图片1为主角，@图片2为场景
→ Reference mapping: @image1 = protagonist, @image2 = scene
```

#### D. `storyboard.ts`（分镜帧生成）

当前去引用正则已支持双语：
```typescript
.replace(/@\s*(?:图片|Image|image)\s*\d+/gi, '')
```
✅ 无需改动。

#### E. `promptPolish.ts`（可灵 Viral）

`【图片1】` / `【TikTok视频】` 这套用于可灵 Omni，建议英文模式改为 `[Image1]` / `[TikTok Video]`。

### 4.4 ⚠️ 坑点：引用标签长度变化

| 引用 | 中文 | 英文 | 差异 |
|------|------|------|------|
| 图片引用 | `@图片1`（4字符） | `@image1`（7字符） | +3 字符 |
| 视频引用 | `@视频1`（4字符） | `@video1`（7字符） | +3 字符 |
| 音频引用 | `@音频1`（4字符） | `@audio1`（7字符） | +3 字符 |

**影响**：如果 prompt 有严格字符限制（如 Seedance 提示词限制），英文引用会多占 3 字符/个。当有 9 张图引用时多 27 字符。需要在 maxTokens 和字符截断处预留 buffer。

## 5. AI 服务层 Prompt 适配

### 5.1 VEO（Compass Video）

| 方面 | 中文模式 | 英文模式 | 注意事项 |
|------|---------|---------|---------|
| 用户输入 | 中文创意 | 英文创意 | — |
| system prompt | 中文指令 | 英文指令 | 需维护两套 |
| polished prompt | 中文输出 | 英文输出 | VEO 对英文更友好 |
| 字数限制 | 50-150 词 | 50-150 words | 英文 1 word ≈ 中文 2 字，实际描述量差异大 |
| 多镜头 | 50-120 词/镜 | 50-120 words/shot | 同上 |

**建议**：英文模式下，VEO prompt 不需要再翻译——直接英文输入、英文润色、英文输出是最优路径。

### 5.2 Imagen（分镜帧生成）

当前 `buildFramePrompts` 已经是中英混合：
```
${lockZh}${core}\n\n[First frame / 首帧] ${en}
```

**改造**：
- 英文模式下去掉 `lockZh`（中文全片画风基准指令），替换为英文版
- `场景描述` fallback → `Scene description`
- 尾部英文指令保持不变

### 5.3 Dreamina / Seedance

即梦的 prompt 格式有严格要求：

| 限制 | 值 | 英文影响 |
|------|-----|---------|
| 提示词最大字数 | 约 2000 字符 | 英文更详细时可能超限 |
| `@图片N` 必须出现在文本中 | 是 | 英文 `@imageN` 需确认即梦 API 是否识别 |
| 结构化格式 | 7 行左右 | 无结构差异，仅语言切换 |

**⚠️ 关键坑点**：

> **即梦 API 是否支持英文引用标签？**  
> 
> 即梦的多模态引用系统底层可能依赖 `@图片N` 的精确字面量来匹配上传的图片位置。如果改成 `@imageN`，即梦 API 可能无法正确关联图片。
> 
> **建议方案**：
> 1. 英文模式下，用户写 `@image1`
> 2. 发送到即梦 API 前，后端做一次 **back-translate**：`@image1` → `@图片1`
> 3. 即梦返回结果后，前端显示仍用英文标签
> 
> ```typescript
> function toSeedanceRefs(prompt: string): string {
>   return prompt
>     .replace(/@image(\d+)/gi, '@图片$1')
>     .replace(/@video(\d+)/gi, '@视频$1')
>     .replace(/@audio(\d+)/gi, '@音频$1');
> }
> ```

### 5.4 可灵（Kling）

可灵 API 对 prompt 语言无严格限制，但：
- **Viral 模板** 的 `【图片1】` 格式可能是可灵 Omni 的协议约定
- 英文模式需确认可灵是否支持 `[Image1]` 或需要保留 `【图片1】`

### 5.5 Lyria / Suno（音乐生成）

- Lyria **原生英文** → 英文模式无需特殊处理
- Suno prompt 限制 500/3000 字符 → 英文通常更短，无风险
- `editorMusicPromptPolish.ts` 的 system prompt 已要求输出英文 → 无需改动
- UI 侧描述（如"描述你想要的配乐风格"）需 i18n

### 5.6 LLM Agent（剪辑 AI 助手）

`editorAgentChat.ts` system prompt 有 "用中文回答"：
- 英文模式改为 "Respond in English"
- Intent router (`editorAgentIntent.ts`) 的 intent 枚举是英文的，无需改

## 6. 后端 API 响应国际化

### 6.1 错误消息

当前大量中文错误消息硬编码在 route handler 中，如：
- `'缺少必填字段 shot'`
- `'AI 返回内容无法解析为合法 JSON'`
- `'请提供 prompt'`

**方案**：

```typescript
// h5-video-tool-api/src/utils/i18nErrors.ts
const errors = {
  MISSING_FIELD: {
    zh: (field: string) => `缺少必填字段 ${field}`,
    en: (field: string) => `Missing required field: ${field}`,
  },
  AI_PARSE_FAILED: {
    zh: 'AI 返回内容无法解析为合法 JSON',
    en: 'Failed to parse AI response as valid JSON',
  },
  // ...
};

export function getErrorMsg(key: string, locale: string, ...args: any[]): string {
  const entry = errors[key];
  const fn = entry?.[locale] ?? entry?.['zh'];
  return typeof fn === 'function' ? fn(...args) : fn ?? key;
}
```

前端请求带上 `Accept-Language` 或 `X-Locale` header，后端据此返回对应语言错误。

### 6.2 数据库默认值

`assets.ai_category` 默认值 `'未分类'` → 需改为 `'uncategorized'`（或保持中文，在前端做映射展示）。

推荐方案：数据库存英文 key（如 `uncategorized`），前端按 locale 显示中文/英文标签。

## 7. 坑点清单 & 风险矩阵

### 7.1 高风险坑点

| # | 坑点 | 风险等级 | 详情 | 缓解方案 |
|---|------|---------|------|---------|
| 1 | **即梦 `@图片N` 是 API 协议约定** | 🔴 高 | 即梦多模态 API 可能只识别 `@图片N`，不接受 `@imageN` | 后端 back-translate；前端显示英文、API 传中文 |
| 2 | **可灵 `【图片1】` 格式依赖** | 🔴 高 | 可灵 Omni 可能将 `【图片1】` 作为多模态绑定标识 | 同上，API 层做格式转换 |
| 3 | **分镜解析器正则** | 🟡 中 | `storyboardParser.ts` 硬编码 `镜头`，英文剧本无法解析 | 改双语正则 |
| 4 | **LLM system prompt 质量** | 🟡 中 | 英文 system prompt 需要单独调优，不能直接翻译中文版 | 每个 prompt 英文版要独立编写+测试 |
| 5 | **中英文 token 成本差异** | 🟡 中 | 同内容英文 token 更少，但 maxTokens 可能不够英文详细描述 | 英文模式 maxTokens 需验证 |
| 6 | **历史数据混合语言** | 🟡 中 | 用户切换语言后，旧剧本/prompt 仍是中文，新的变英文 | 不翻译历史数据，标注 `contentLanguage` 字段 |
| 7 | **loading copyPool 全是中文** | 🟢 低 | `copyPool.ts` 有 50+ 条影院式 loading 文案 | 维护一套英文 copyPool |
| 8 | **toLocaleString 硬编码** | 🟢 低 | 日期格式 `zh-CN` 硬编码 | 改为读 i18n.language |

### 7.2 容易忽略的坑点

| # | 坑点 | 详情 |
|---|------|------|
| 9 | **中文标点 vs 英文标点** | 解析器/正则中的 `。`、`，`、`：`、`｜`、`（）` 在英文模式需要同时匹配英文标点 `.`, `,`, `:`, `|`, `()` |
| 10 | **Prompt 模板文件** | `config/prompt-templates/*.json` 中的 `systemPromptSuffix` 全是中文（如猫后宫、Viral 舞蹈模板），英文模式需要对应英文版本 |
| 11 | **名称注入逻辑** | `productionAssets.ts` 的 `tryInjectNameWithTag` 用中文语法（如跳过"名字+的"避免误匹配），英文需要不同的命名启发式 |
| 12 | **Seedance 结构化 prompt 区段标签** | `【主体】`、`【环境】`、`---运动---` 这些中文区段名，英文改为 `[Subject]`、`[Environment]`、`--- Motion ---`。但 Seedance 是否对区段名有解析依赖？ |
| 13 | **TikTok caption 字数规则** | 当前中文 15-45 字、英文 30-120 字符已在 `TIKTOK_RULES` 中分别规定，但如果 contentLocale=en 时，LLM 可能仍产出中文 caption（如果 system prompt 没切好） |
| 14 | **multer 文件名编码** | `multipartFilename.ts` 处理 CJK garbling，英文文件名不会出此问题，但双语混用时仍需保留此逻辑 |
| 15 | **`productionTypes.ts` 预设标签** | `CHARACTER_STATE_PRESETS` 的 labels 和 `statePrompt` 是中文（如"微笑"、"战斗姿态"），需英文版本 |
| 16 | **素材标签系统** | `assetTaggingService.ts` 的 system prompt 要求中文分类标签，英文模式需改为英文标签 |
| 17 | **asset 默认分类 `未分类`** | DB 默认值是中文字面量，需做数据兼容 |
| 18 | **Step 名称** | QuickFilm 的步骤如 `分析故事结构`、`生成角色设定` 是中文，需 i18n |
| 19 | **`joinStill` / `joinMotion` 的区段头** | 如 `【主体】`、`【光线】` 等，英文版对应？ |
| 20 | **短剧模板的场次格式** | `场次1-订婚宴` → `Scene 1 - Engagement Banquet`，解析逻辑是否依赖 "场次" 关键字？ |

### 7.3 Seedance / 即梦特有风险

| 风险 | 说明 | 建议 |
|------|------|------|
| **英文 prompt 字符数限制** | 即梦 prompt 上限约 2000 字符。英文表达通常比中文长 30-50%（因为中文信息密度高） | 英文模式下 composer 的 `maxTokens` 需增加，truncation 阈值需调整 |
| **英文 prompt 生成质量** | Seedance 主要面向中文市场优化，英文 prompt 的视频生成质量可能不如中文 | 需要 A/B 测试对比；考虑 "英文界面 + 中文 prompt" 选项 |
| **多模态引用转换的边界条件** | `@image10` 转 `@图片10`：双位数索引、连续多个引用、引用前后紧挨文字 | 正则需严格匹配 `@image(\d+)` 而非简单字符串替换 |
| **Dreamina normalizedName 截断** | 当前 `slice(0, 16)` 按 JS string length（UTF-16 code units）。中文名"林晓风"=3 个 code unit，英文名 "Alexander Hamilton" = 19 个 code unit | 英文模式下名称截断长度需加倍至 32 |
