## 三、经验教训

### 3.1 高频致命类（出现 2+ 次的同类问题）

| 类别 | 出现次数 | 典型事件 | 根因 |
|------|---------|---------|------|
| 鉴权链路不完整 | 3 次 | 视频 403 黑屏、导出 401、多用户串扰 | 改一端不改另一端，缺乏端到端验证 |
| 数据丢失/刷新即丢 | 4 次 | 剪辑器丢项目、分镜视频丢失、submitId 丢失、版本切换不同步 | 前端状态依赖内存，异步 ID 未持久化 |
| 即梦并发/重试 | 3 次 | 1310 超限、信号量设置过高、重启后残留任务 | 对外部 API 限制认知不足，逐步修补 |
| 服务器路径/配置 | 3 次 | Windows 路径、loadEnv 错误、PYTHON_EXE 不存在 | 本地 .env 覆盖服务器 .env |

### 3.2 沉淀为规则的教训（feedback.md）

1. 禁止修改后端底层服务文件
2. `npm run build` 必须零错误
3. 不硬编码 API Key
4. 不在没有 planner-spec 的情况下开始 build
5. 每次改动后必须更新 PRODUCT.md
6. **数据清理函数必须按字段名白名单，禁止全量模式匹配**（源于 stripBase64 误删用户头像）
7. **修鉴权 bypass 时必须同步检查路由层的用户校验**（源于视频 403 黑屏）

## 六、后续优化建议

### 6.1 建立"变更影响检查表"

每次修改前先填写：

```
□ 涉及哪些端（前端/后端/服务器）？
□ 涉及的数据流完整路径是什么？
□ 鉴权改动：是否同步检查了 auth 中间件和路由内部？
□ 数据清理：使用字段名白名单了吗？
□ 外部 API：并发/超时/重启恢复都考虑了吗？
```

### 6.2 合并小版本，减少"修了又修"

```
❌ v0.24 开并发 → v0.25 改回 → v0.34 修超时 → v0.38 加重试
✅ v0.24 即梦并发机制（含边界测试 + 重试 + 友好提示）一次到位
```

目标：**每个版本至少存活 2 天不需要 hotfix**。

### 6.3 外部 API 集成的"完整测试矩阵"

对接即梦/Compass/Suno 等外部服务时，必须覆盖：

```
□ 正常路径（1 个请求成功）
□ 并发路径（N 个同时请求）
□ 超限路径（被限流时的降级）
□ 超时路径（10 分钟无响应）
□ 重启恢复（服务重启后旧任务状态）
□ 鉴权失效（token 过期/未登录）
```

### 6.4 视频数据生命周期统一管理

建立统一的 `VideoLifecycleService`：

```
生成（即梦/VEO）→ 下载落盘 → 路径注册 → 鉴权放行 → URL 构建 → 前端播放
```

每个环节有明确的状态和错误处理，而非在每个功能点各自处理。

### 6.5 版本号规范化

```
当前：v0.11 → v0.39（29 个版本/天，有重复）
建议：按语义化版本，重大功能 minor++，Bug Fix patch++
      同一功能的开发+修复合并为一个版本
```
