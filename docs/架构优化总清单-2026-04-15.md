## 🚨 紧急修复（线上已有问题）

### 紧急-1：Nginx 服务的不是最新前端版本

**来源**：Cursor 现场发现  
**现象**：服务器上存在 3 份前端副本，Nginx root 指向的不是最新部署目录：
```
/var/www/qas-h5/              ← Nginx 实际服务（Apr 14 19:11）⚠️ 旧
/home/ubuntu/qas-h5/h5/       ← 中间版本（Apr 14 16:55）
/home/ubuntu/qas-h5/frontend/ ← 最新部署（Apr 14 21:48）✅ 未被使用
```
**修复**：把 Nginx 配置 root 改为 `/home/ubuntu/qas-h5/frontend`，并清理冗余副本。

---

### 紧急-2：SQLite 数据库文件落在 output/ 目录

**来源**：Cursor 现场发现  
**现象**：`/home/ubuntu/qas-h5/output/assets.db` 数据库混在视频输出目录里，若误清理 output/ 将导致数据丢失。  
**修复**：迁移至 `/home/ubuntu/qas-h5/api/db/`，检查 `apiDataDir.ts` 相关路径配置。

---

### 紧急-3：SERVER_PASSWORD 存在服务器的 .env 中（安全隐患）

**来源**：Cursor 现场发现  
**现象**：服务器 `/home/ubuntu/qas-h5/.env` 包含 `SERVER_PASSWORD` 字段（服务器自身的 SSH 登录密码），一旦后端有任意文件读取漏洞，SSH 密码即泄露。  
**修复**：从服务器端 .env 删除该字段（`SERVER_PASSWORD` 只供本地部署脚本使用）。

---

### 紧急-4：API 服务端口 3001 对外暴露

**来源**：Cursor 现场发现  
**现象**：`ss -tlnp` 显示 Node 进程监听 `0.0.0.0:3001`，外网可绕过 Nginx 直接访问 API。  
**修复**：在 `src/index.ts` 的 `app.listen` 加绑定地址参数改为 `'127.0.0.1'`。

## 🔴 P0：核心架构问题（两周内）

### P0-A：建立最小 CI/CD 闭环 + 版本追踪

**来源**：两份报告共同指出  
**现状**：手动 tsc → 手动 SFTP → 手动 pm2 restart，任意步骤漏做都导致三端不一致（紧急-1 就是这个的后果）。  
**建议**：
1. 统一 `build → package → deploy` 脚本（或 GitHub Actions）
2. 构建时生成 `build-info.json`（commit sha、branch、构建时间）
3. 后端新增 `/api/system/version` 接口返回运行版本
4. 产物命名带 sha，服务器保留最近 3 版回滚包
5. 部署脚本统一上传目标目录，消除多副本混乱

**验收**：`/api/system/version` 返回 sha 与构建时间；线上改动可追溯到具体 commit。

### P0-C：统一路径与存储抽象层（storageResolver）

**来源**：两份报告共同指出  
**现状**：服务器上存在 6 个分散数据路径：
```
api/output/     api/uploads/    api/quickfilm/
api/.data/      api/data/       (root)/output/  ← 有数据库！
```
业务代码中散落路径拼接逻辑。  
**建议**：
- 新建 `src/infra/storage/resolver.ts`，业务层只传资源类型 + 业务参数，不直接拼路径
- 启动时做目录自检，不存在则自动创建或明确报错
- 全局 grep 替换硬编码路径

**验收**：grep 业务代码不再出现 `/home/ubuntu` 绝对路径（resolver 层除外）。

## 🟡 P1：可维护性提升（2-4 周）

### P1-E：长任务队列化（同步阻塞→异步 Job）

**来源**：两份报告共同指出（对应 TASK-04）  
**现状**：`POST /generate-multishot` 串行生成所有镜头，5 镜头可能挂 HTTP 连接 10 分钟；PM2 已累计重启 17 次，可能与此相关。  
**建议**：引入 BullMQ 或 Bee-Queue，API 仅 enqueue 返回 `jobId`，Worker 负责长任务，支持 retry/backoff/并发限流。

---

### P1-F：配置治理（12-Factor App）

**来源**：Openclaw 提出，Cursor 补充  
**现状**：
- `.env.example` 不完整，缺少默认值说明
- 启动时没有 env 校验，缺失关键变量时服务静默异常
- 服务器 .env 与本地 .env 字段集不对称（服务器多了 `SERVER_PASSWORD` 等本地专用字段）

**建议**：
- 完善 `.env.example`（必填/可选/默认值三列注释）
- 使用 `zod` 或 `envalid` 在启动时校验关键变量
- 区分"本地开发专用"和"服务器部署必需"两套变量集

**校验字段最低要求**：`COMPASS_API_URL`、`COMPASS_API_KEY`、`API_DATA_DIR`、`JWT_SECRET`

---

### P1-G：外部依赖状态机标准化（Dreamina / Compass）

**来源**：Openclaw 提出  
**现状**：Dreamina 未登录、Compass 429、超时三类异常下，前端可能误判为"已完成"。  
**建议**：
- 统一任务状态枚举：`queued / running / succeeded / failed / timeout / canceled`
- 新建 `src/domain/job-status.ts`
- 所有外部调用返回统一结构：`{ status, errorCode, errorMessage, startedAt, endedAt }`
- 前端只依据后端状态机渲染，不做客户端猜测

---

### P1-H：API 分层重构（Router → Service → Infra）

**来源**：Openclaw 提出  
**现状**：routes/ 文件直接包含业务逻辑和外部调用；`/api/editor` 有 6 个 router 共享同一前缀（易路径冲突）。  
**建议**：
```
routes/   ← 参数校验、鉴权、响应码（不含业务逻辑）
services/ ← 业务编排（现状基本合理，继续细化）
infra/    ← 文件系统、第三方 API、CLI 调用（新增）
domain/   ← 类型定义、状态机（新增）
```
优先重构高风险模块：`video.ts`（690 行）、`editorAssets.ts`（532 行）。

---

### P1-I：前后端类型同步机制

**来源**：Cursor 发现  
**现状**：`editor/timelineSchema.ts`（后端）和 `editor/types/timeline.ts`（前端）靠注释约定手动同步，无编译保证。  
**建议**：将共享类型提取到 `shared-types/` 包（monorepo 或 npm workspace），前后端共同引用；或至少加 JSON Schema 比对脚本。

### P2-L：前端状态管理统一（React Query）

**来源**：两份报告共同指出  
**现状**：多层 Context 嵌套（4 层 Provider）+ 各页面独立 `useState`，随规模增长难以维护；轮询逻辑分散在多个 hook 中。  
**建议**：引入 React Query（TanStack Query）统一管理服务端状态，覆盖 loading/error/retry；长任务统一轮询策略与停止条件。
