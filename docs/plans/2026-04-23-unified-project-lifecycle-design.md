# Unified Project Lifecycle Design

更新时间：2026-04-23

## 目标

统一优化视频剪辑与高级制片两端的项目生命周期，解决“进入页面后不断自动生成未命名项目”的问题，让正式项目列表只承载值得被再次找到的工作。

本次设计的核心目标有两条：

1. 以后不再继续新增“未命名项目”污染正式项目列表
2. 为历史上的未命名项目提供批量治理入口，而不是要求用户逐个清理

## 问题判断

当前两端都存在“工作刚开始就过早创建正式项目”的问题：

- 剪辑器会在用户进入后很快形成可自动保存状态，默认名又是“未命名剪辑项目”，导致列表被弱价值项目占满
- 高级制片会持续把当前状态同步到服务端，只要标题为空，后端就会回填成“未命名项目”，所以污染更明显
- 两端的项目列表治理能力不一致，导致用户即使知道问题，也没有高效清理手段

这不是命名文案问题，而是项目生命周期设计问题。系统把“临时工作态”过早升级成了“正式项目”。

## 设计原则

### 1. 草稿不等于项目

用户刚进入页面时，应该先处于“临时草稿态”。临时草稿可以存在于页面状态或本地存储中，但不应立即进入正式项目列表。

### 2. 正式项目必须可识别

一旦内容值得长期保留，系统可以要求转正；而“转正”必须带名字。正式项目列表里不应再出现新的空标题或“未命名项目”。

### 3. 两端规则一致

剪辑器和高级制片必须使用同一套心智模型：

- 先草稿
- 形成有效内容
- 强制命名
- 转正入库

### 4. 历史治理要批量化

已经存在的未命名项目不能再靠用户逐条处理。至少要提供“一键治理未命名项目”的批量入口。

## 统一项目生命周期

### 阶段 A：临时草稿态

进入剪辑器或高级制片时，系统先创建临时工作区，但不创建正式项目记录。

此时的特征：

- 不进入“我的项目”正式列表
- 不生成新的正式项目 id
- 可以编辑、上传、生成、试验
- 可以保留本地续写能力，但不应污染服务端正式项目列表

### 阶段 B：达到转正阈值

当用户第一次产生“值得保存”的内容时，系统判断草稿已经具备转正价值。

剪辑器转正阈值：

- 时间轴出现任意视频 clip
- 添加字幕或文字片段
- 生成 BGM
- Agent 真正改动了时间轴
- 加入素材并形成可编辑时间轴

高级制片转正阈值：

- 生成剧本大纲
- 生成角色/场景/道具
- 生成分镜
- 上传参考图并形成有效制作内容
- 项目中出现稳定可复用的制作结果

### 阶段 C：强制命名转正

一旦达到转正阈值，系统必须拦截一次命名。

规则：

- 用户必须填写项目名
- 不命名就不能转正
- 命名成功后才真正创建或首次写入正式项目
- 之后继续沿用现有自动保存

这次不采用“自动给个默认可识别名字然后继续”的策略，而是直接执行强命名，因为目标就是清除新未命名项目的入口。

## 两端具体行为

### 剪辑器

当前核心位于：

- `h5-video-tool/src/editor/hooks/useTimelineState.ts`
- `h5-video-tool/src/pages/EditorWorkbench.tsx`
- `h5-video-tool-api/src/routes/editorProjects.ts`

改造后：

- 初始空白态不再默认携带“未命名剪辑项目”
- 当草稿第一次达到转正阈值时，暂停自动保存并弹出命名层
- 命名成功后才调用正式项目保存接口
- 显式点击“+ New”仍然允许直接输入名字后进入一个新正式项目

### 高级制片

当前核心位于：

- `h5-video-tool/src/pages/ProductionWizard.tsx`
- `h5-video-tool/src/studio/ProductionWizardShell.tsx`
- `h5-video-tool-api/src/routes/productionPersist.ts`

改造后：

- 允许本地草稿继续保留，但在未命名且未转正前不向服务端创建正式项目
- 当草稿达到转正阈值时，弹出强制命名层
- 命名成功后才开始首次服务端保存并分配正式项目 id
- 已有 URL 指向的正式项目继续照常加载

## 历史未命名项目治理

这轮不自动改写旧项目，也不偷偷隐藏它们，而是提供显式批量治理入口。

治理入口目标：

- 批量识别未命名项目
- 批量删除“明显空项目”
- 批量为“有内容但未命名”的项目生成可识别标题

批量治理策略：

- 编辑器在项目管理弹窗中增加“治理未命名项目”入口
- 高级制片在项目列表弹窗中增加同名入口
- 点击后系统批量扫描所有未命名项目
- 空项目直接删除
- 非空项目自动重命名为建议名
- 最终给出“删除 X 个 / 重命名 Y 个 / 跳过 Z 个”的结果反馈

## 建议命名策略

### 剪辑器

优先级：

1. 来源制片标题 `sourceProductionTitle`
2. 第一条素材名或原始文件名
3. 频道化兜底名，如 `剪辑-0423-1530`

### 高级制片

优先级：

1. `project.meta.title` 中已有有效标题
2. 故事摘要 / 剧本大纲中的首要主题词
3. 兜底名，如 `高级制片-0423-1530`

命名目标不是“完美创意标题”，而是“未来能找回来”。

## 后端防回归策略

即使前端做了拦截，后端也需要增加最小保护：

- 编辑器创建新项目时，如果是首次创建且名字为空，则拒绝创建
- 高级制片首次服务端保存时，如果项目标题为空，则拒绝创建

这样即使未来有其他前端入口绕过 UI，也不会重新长出新未命名项目。

## 验收标准

- 进入剪辑器不会自动新增未命名正式项目
- 进入高级制片不会自动新增未命名正式项目
- 两端第一次达到转正阈值时都会强制命名
- 命名完成后项目才会进入正式列表
- 两端都提供“治理未命名项目”批量入口
- 治理入口可以批量删除空未命名项目，并批量重命名有内容未命名项目

## 实施边界

本轮不做：

- 历史项目的自动后台迁移
- 复杂的批量手工勾选与逐项编辑 UI
- 项目收藏、归档、置顶
- 基于用户行为的高级智能命名模型

本轮优先做：

- 生命周期收口
- 强制命名转正
- 最小批量治理入口
