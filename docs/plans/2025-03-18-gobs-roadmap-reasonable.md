# Gobs 项目时间线与资源规划（调整版）

> **更新日期**: 2025-03-18  
> **目的**: 基于当前 QAS 基线，将原规划调整为更可落地、可交付的版本。

---

## 一、当前基线（2025-03 现状）

| 模块 | 状态 | 缺口 |
|------|------|------|
| 视频生成 | 本地跑通，H5 未打通 | 缺分镜 API、Seedance API、Compass/VEO 已部分接入 |

| 分发 | Skills 可闭环，H5 未打通 | 缺 GeeLark 发布 API、账号管理 UI |

| 本地化与鉴权 | 无 | 全新开发 |

---

## 二、原规划 vs 合理性问题

| 维度 | 原规划 | 问题 |
|------|--------|------|
| **1.0 时间** | 2 人，~6 周（3 月中→4 月底） | 6 周内完成：3 模板 + 40% 成片率 + 剪辑 + 40 设备 + 一键发布 + 水军 + 养号 + 本地化 + 鉴权，明显超载 |
| **1.0 成片率** | 40%+ | 需大量 prompt 工程与验证，与模板数量强耦合，时间成本高 |
| **账号规模** | TH+ID 各 20+ | 采购、上架、登录、养号周期不可压缩 |
| **水军/养号** | 未定义 | 功能边界不清，可能有合规风险 |
| **Crowley** | 2.0 闭环 | 需确认是外部系统还是内部模块 |
| **人效** | 2→3→5 线性加人 | 新人 ramp-up、沟通成本，产出非线性 |

## 三、调整后规划（推荐）

### 总体思路

- **MVP 优先**：先打通端到端，再堆功能。
- **分批交付**：1.0 做「可推广的最小可用产品」，2.0/3.0 按数据与反馈迭代。
- **人时预留**：为联调、测试、部署留 20–25% buffer。

---

### Gobs 1.0 — 4 月底（2 人，约 6 周）

**目标**：Marketer 可在 TH/ID 用 Gobs 完成「生成 → 简单剪辑 → 分发」的闭环。

| 功能 | 范围 | 说明 |
|------|------|------|
| **视频生成** | 1–2 个成熟模板 | 先保证 1 个模板成片率 ≥35%，再扩展第 2 个；40% 放到 2.0 优化 |
| **剪辑** | 片头/片尾 + 基础字幕 | 支持上传/选择模板，不支持复杂时间线；本地化字幕为「批量替换文案」 |
| **内容分发** | TH 10 设备 + ID 5 设备起 | 先打通，再在 2.0 扩展到 20+；一键发布 + 任务状态查询 |
| **水军/养号** | 1.0 不纳入 | 放到 2.0，需单独产品定义与合规 review |
| **本地化** | 双语（TH/ID）UI | 文案与配置支持 TH/ID 切换 |
| **鉴权** | 简易登录 | 邮箱/内部 SSO 等，不做完整 RBAC |

**交付物**：可用的 H5 工具 + 1 份使用说明 + 部署文档。

---

### Gobs 2.0 — 6 月底（3 人，约 8 周）

**目标**：闭环 idea → content → performance，支撑规模化运营。

| 功能 | 范围 | 说明 |
|------|------|------|
| **Crowley 闭环** | 待确认 | 若为外部系统，需明确 API 与数据流；若为内部，需定义埋点与报表 |
| **剪辑增强** | 时间线 + 多轨 | 支持多段素材、简单转场、更灵活的字幕 |
| **账号规模** | TH 20+ / ID 20+ | 补齐设备与 TT/FB 账号，完善养号流程 |
| **水军/养号** | 基础能力 | 需产品 spec：触发条件、执行逻辑、风控与合规 |
| **成片率** | 目标 40%+ | 基于 1.0 数据迭代 prompt 与模板 |

**前提**：1.0 已稳定运行，有真实使用数据可分析。

---

### Gobs 3.0 — 12 月底（4–5 人，约 6 个月）

**目标**：Agent 模式，用自然语言驱动全流程。

| 功能 | 范围 | 说明 |
|------|------|------|
| **Agent 模式** | 指令 → 自动执行 | 如「为 TH 生成 3 条浪人视频并发布到 TikTok」 |
| **智能大脑** | 理解意图 + 选模板 + 调参数 | 依赖 1.0/2.0 的数据与规则积累 |
| **规模与稳定性** | 多地区、多账号 | 支持更多地区与平台，运维与监控 |

**资源**：建议 4 人即可启动；第 5 人根据 2.0 进度与 3.0 复杂度再定。

## 五、关键假设与风险

| 项目 | 假设 | 风险 |
|------|------|------|
| 设备与账号 | GeeLark 有 TH/ID 可用资源，采购周期 ≤2 周 | 供应链或配额不足会拖累 1.0 |
| Crowley | 2.0 前有明确对接方式（API/数据格式） | 若依赖外部排期，可能影响 2.0 时间 |
| 水军/养号 | 有清晰的产品定义与合规绿灯 | 边界不清会导致返工 |
| 成片率 40% | 依赖 Seedance/VEO 能力 + prompt 迭代 | 需预留 2–3 周专门调优 |

## 六、待确认事项

1. **Crowley**：是内部模块还是外部系统？需要哪些接口与数据？
2. **水军/养号**：具体指什么能力？自动化评论、点赞、关注，还是账号健康度维护？
3. **1.0 设备目标**：TH 10 + ID 5 是否能满足首批推广？还是必须 20+20？
4. **鉴权**：是否有统一 SSO/IdP？还是先做简单账号密码？
