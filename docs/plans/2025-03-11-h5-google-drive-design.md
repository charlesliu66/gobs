## 二、Google Drive 集成方案

### 2.1 认证流程

| 方式 | 说明 | 适用 |
|------|------|------|
| **Google Identity Services (GIS)** | 前端 OAuth 弹窗，用户授权 `drive.readonly` | H5 推荐 |
| **OAuth 2.0 完整流程** | 后端做 redirect，前端拿 token | 需后端时 |

**H5 推荐**：使用 [Google Identity Services](https://developers.google.com/identity/gsi/web) + `drive.readonly` scope，用户点击「连接 Google Drive」→ 弹窗授权 → 获取 access_token。

### 2.2 检索逻辑（与本地 video-pipeline 一致）

```
输入：关键词列表 [血坦, 武器, 奔跑]
目标：用户整个 Drive（全量检索）

Drive API: files.list
  - q: "mimeType contains 'image/' or mimeType contains 'video/'"
  - q: "name contains '血坦' or name contains '武器' or name contains '奔跑'"
  - 不限定 folderId，全 Drive 搜索
  - 返回：id, name, thumbnailLink, webContentLink
```

**排序**：匹配关键词数量多的优先；同等情况按 name 排序。

### 2.3 文件获取

| 场景 | 方式 |
|------|------|
| **预览** | 用 `thumbnailLink` 或 `webContentLink` 直接展示 |
| **上传到 Seedance** | 需下载文件内容。Drive API `files.get(id, alt=media)` 返回二进制，前端可转 Blob；若后端调用 Seedance API，则后端负责下载并上传 |

---

## 2.4 成片保存到用户电脑

**可以**。生成后的视频可通过以下方式让用户保存到本地：

| 方式 | 说明 |
|------|------|
| **A. 直接下载** | 后端返回视频 URL 或流，前端用 `<a download>` 或 `fetch + Blob + URL.createObjectURL` 触发浏览器下载，用户选择保存位置 |
| **B. 保存到用户 Drive** | 用户已授权 Drive，后端用 Drive API `files.create` 将成片上传到用户指定文件夹，用户再从 Drive 下载到电脑 |
| **C. 两者都支持** | 提供「下载到电脑」和「保存到 Drive」两个按钮 |

**推荐**：A 为默认（一键下载到电脑），B 为可选（便于云端备份、多设备同步）。

## 三、前后端分工

### 3.1 纯前端可完成

| 能力 | 实现 |
|------|------|
| Prompt 输入 | 表单 |
| 关键词提取 | 简单规则（2 字及以上实体词，去助词）或调用轻量 API |
| Google Drive 连接 | GIS OAuth + Drive API（需 CORS 或代理） |
| Drive 文件搜索 | `files.list` + q 参数 |
| 素材预览、排序、确认 | 前端 UI |
| 复制 prompt | `navigator.clipboard` |

### 3.2 需要后端

| 能力 | 原因 |
|------|------|
| 分镜脚本生成（LLM） | 需调用 OpenAI/Claude 等，API Key 不能暴露 |
| Seedance API 调用 | 若自动生成视频，需 API Key 与计费 |
| Drive API 代理（可选） | 若遇 CORS，可用后端代理 Drive 请求 |
| 成片下载 / 保存到 Drive | 后端提供下载链接或代理 Drive 上传 |
| 批量发布（GeeLark） | 需后端调用 GeeLark API，API Key 不可暴露 |

## 四、MVP 推荐方案

### 方案 A：轻量 MVP（前端为主）

```
H5 页面
  → 用户输入 prompt
  → 前端提取关键词
  → 用户点击「连接 Drive」→ OAuth → 搜索匹配文件
  → 用户选择/排序素材
  → 前端按 video-director 规则生成 prompt（规则引擎，无 LLM）
  → 输出：复制 prompt + 跳转 Seedance，用户手动上传素材
```

**优点**：实现快，无后端，无 API 费用  
**缺点**：分镜质量依赖规则；需用户手动在 Seedance 上传素材

### 方案 B：完整体验（含后端）

```
H5 + Node.js 后端
  → 同上，但分镜生成走 LLM API
  → 后端下载 Drive 文件 → 调用 Seedance API 生成视频
  → 返回成片链接或下载
```

**优点**：全自动，体验好  
**缺点**：需服务器、Seedance API 费用、Drive 下载需后端（或临时 URL）

## 六、Skills 复用方案（回答「能否调用 skills」）

### 6.1 直接调用 Cursor Skill 工具？

**不能**。Cursor 的 Skill 工具是 Cursor 内部机制，H5 后端无法直接调用。

### 6.2 间接复用：把 Skills 当作 LLM 上下文

**可以**。Skills 本质是「结构化指令 + 规则」，可读入后作为 LLM 的 system prompt / context：

```
后端加载以下文件内容：
  - video-director/SKILL.md      → 工作流、分镜格式、@图片N 映射规则
  - storyboard-studio/SKILL.md   → 镜头类型、构图、运镜、时间码规范
  - seedance-rules.md            → 禁用词、替换规则、审核清单

拼接为 LLM system prompt：
  "你是一个视频分镜专家。请严格遵循以下规范：[上述内容]
   用户输入：[用户 prompt]
   已选素材及映射：[@图片1=xxx.png, @图片2=yyy.jpg]
   请输出符合 Seedance 规范的最终 prompt。"
```

### 6.3 实现方式

| 方式 | 说明 |
|------|------|
| **A. 打包进后端** | 将 skills 目录复制到项目，后端启动时读取，构建 prompt |
| **B. 远程加载** | Skills 放 Git/OSS，后端按需拉取（可版本管理） |
| **C. 规则引擎 + LLM 混合** | seedance-rules 用代码实现（替换表），分镜创意用 LLM + storyboard/video-director 规范 |

**推荐**：A 或 C。seedance 禁用词替换用代码做更稳，分镜生成用 LLM + skill 文案。

## 七、发布模块（批量发布到社媒）

### 7.1 能力概述

用户选择已生成的视频，勾选目标平台（INS、TikTok、YouTube、Facebook），一键批量发布。

### 7.2 技术方案（复用 Quality at Scale 设计）

| 组件 | 说明 |
|------|------|
| **GeeLark 云手机** | 多设备独立指纹，每台可登录不同社媒账号，通过 API 批量发布 |
| **后端集成** | 成片 → 上传到 GeeLark 素材库 → 调用 Automation API → 分发到各平台 |
| **用户侧** | 用户需在 GeeLark 中预先配置云手机并登录各社媒账号；或由平台提供托管账号（需商业模型） |

### 7.3 用户流程

```
成片列表
  → 用户勾选要发布的视频
  → 勾选目标平台（INS / TikTok / YouTube / Facebook）
  → 填写文案、标签（可选，可各平台不同）
  → 点击「批量发布」
  → 后端调用 GeeLark API 执行
  → 展示发布状态 / 链接
```

### 7.4 前置条件

- 用户需有 **GeeLark 账号**，并已创建云手机、登录社媒
- 或：平台提供 **GeeLark 集成**，用户绑定自己的 GeeLark API Key（敏感，需安全存储）
- 或：平台自建发布能力（对接各社媒官方 API，复杂度高）

### 7.5 实现优先级

| 阶段 | 内容 |
|------|------|
| **Phase 1** | 成片下载到电脑 + 保存到 Drive |
| **Phase 2** | 发布模块：支持用户绑定 GeeLark，批量发布到选定平台 |
