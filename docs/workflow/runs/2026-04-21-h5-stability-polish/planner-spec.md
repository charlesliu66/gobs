# Planner Spec - h5 stability polish

> Run: `2026-04-21-h5-stability-polish`
> Gate 1 产物，覆盖当前线上 H5 的首批稳定性优化。

---

## 背景

2026-04-21 的项目审查中，线上 H5 仍存在 4 个直接影响可用性的坑点：

1. `batch-jobs` 成片链接没有统一补 FAT/JWT，真实浏览器场景下预览、下载、导入剪辑存在 401 风险
2. `batch-jobs` SSE 一旦断线就静默失联，长任务场景用户会误以为系统卡死
3. 剪辑器顶部“来自制片项目”回跳链接还指向旧路由 `/studio/wizard`
4. 侧边栏版本号仍是硬编码 `GOBS v0.1`，与实际线上运行版本脱节

本轮目标不是做大重构，而是先把这 4 个问题补齐，为下一轮架构收口降低噪音。

---

## SP-01：batch-jobs 媒体鉴权链路收口

### 目标
保证批量任务成片的三条前端消费链路都统一走“绝对 API URL + FAT/JWT query”：
- 看板内预览
- 看板内下载
- 历史页导入剪辑

### 方案
- 复用现有 `appendFileAccessToken()`，不要重复发明鉴权逻辑
- 同时用 `absoluteApiUrl()` 将相对路径转为真实 API 地址，兼容 `VITE_API_BASE_URL`
- 只在消费侧补齐，不改后端 `videoUrl` 字段格式，降低影响面

### AC
| # | 场景 | 预期 |
|---|---|---|
| AC-1 | 看板展开已完成任务 | `<video>` 可正常播放 |
| AC-2 | 点击下载视频 | 新标签页或下载链路不再 401 |
| AC-3 | 历史页导入批量任务视频 | 能成功拉取 Blob 并导入素材库 |

### 风险
- 老用户本地没有 `gobs_fat`：通过 `appendFileAccessToken()` 内部懒加载和 JWT 旁路兼容

---

## SP-02：batch-jobs SSE 自动重连

### 目标
`BatchJobsBoard` 与全局任务流在网络抖动、Nginx reload、后端短暂重启后都能自动恢复连接。

### 方案
- 保留现有 `EventSource` 模式
- 新增前端指数退避重连
- `BatchJobsBoard` 增加最小可见的“重连中”状态提示
- 不引入新的状态管理库，不改后端 SSE 协议

### AC
| # | 场景 | 预期 |
|---|---|---|
| AC-1 | SSE 正常建立 | 看板显示“实时同步” |
| AC-2 | 连接报错 | 前端自动关闭旧连接并按退避策略重连 |
| AC-3 | 重连期间 | 看板可见“重连中”提示，不再静默失联 |
| AC-4 | 重连成功后收到新事件 | 任务状态继续更新 |

### 风险
- 若后端长期不可达，会持续重试：通过最大退避时间限制频率，避免过度打点

## SP-03：剪辑器回跳制片路由修复

### 目标
从剪辑器顶部入口能准确回到对应的高级制片项目。

### 方案
- 把旧路由 `/studio/wizard?project=...` 改为 `/studio/production?projectId=...`
- 与 `ProductionWizard` 当前 query 约定保持一致

### AC
| # | 场景 | 预期 |
|---|---|---|
| AC-1 | 有来源制片项目的剪辑工程 | 顶部回跳入口存在 |
| AC-2 | 点击回跳 | 打开正确的制片项目，不落到空白页 |

---

## SP-04：真实运行版本展示

### 目标
侧边栏底部显示当前运行中的真实版本信息，便于用户和开发确认线上环境。

### 方案
- 前端在 `Layout` 挂载后请求 `/api/system/version`
- 优先显示 `branch@commitShort`，缺失时退化为 `branch@commitSha[0:7]`
- 请求失败时保留简短 fallback，不阻塞主界面

### AC
| # | 场景 | 预期 |
|---|---|---|
| AC-1 | 接口返回版本信息 | 底部显示真实分支与短 commit |
| AC-2 | 接口失败 | 页面正常，显示简短 fallback |
