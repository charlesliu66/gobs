# Campaign Creative Agent 下一阶段设计稿

> 日期：2026-05-06
> 状态：已确认方向，待按新北极星拆解实施

---

## 1. 这次重想的核心结论

平台的主用户不应该再默认是剪辑师或制片，而应该明确切到市场和运营。

这意味着产品的默认形态不该继续围绕“更多专业控制项”展开，而要围绕“给定 brief 后，系统持续稳定地产出更能跑的内容”展开。

换句话说，平台的本体不是一个更强的视频工具，也不是一个 editor 入口，而应该是：

> 一个从 campaign brief 出发，会思考、会生产、会分发、会记住人工反馈，并逐步减少人工干预的 campaign growth agent。

---

## 2. 最终产品承诺

后续所有 run 都应被同一句产品承诺约束：

> 给定 brief 后，系统持续稳定地产出更能跑的内容。

这句话背后有四个必须同时成立的条件：

1. `有大脑`
   系统不是机械生成，而是能判断这轮 campaign 该产什么、为什么这么产。

2. `能产不同类型素材`
   目标不是只出一条视频，而是按 campaign 目标产出合适的素材组合。

3. `有分发能力`
   终点不能停在 editor 或 asset preview，系统要把内容真正送进分发。

4. `人工干预越来越少`
   前期允许 review 和人工纠偏，但长期必须朝自动托管演进。

---

## 3. 目标受众与产品主语

### 3.1 主受众

- 市场负责人
- 运营 / UA 同学
- 需要快速推进 campaign，但不想深度操作视频工具的人

他们真正关心的是：

- 这轮 campaign 要做什么内容
- 系统为什么这么安排
- 哪些需要我处理
- 哪些已经发出去了
- 下轮如何更聪明

### 3.2 次受众

- 剪辑师
- 制片
- 需要深度微调素材的专业同学

他们不是消失，而是不再定义默认体验。专业能力应保留，但下沉到 `Advanced Studio`。

### 3.3 产品主语切换

用户不是来管理工具的，也不是来管理单条素材的，而是来推进一个 campaign 持续跑出更好的结果。

因此平台的核心对象不该是 `video`、`project` 或 `editor session`，而应该是 `Campaign`。

---

## 4. 推荐路线

### 4.1 三条候选路线

1. `轻量 Copilot`
   系统给策略、给素材建议、给分发建议，但每一步都要人确认。

2. `Review-first Agent`
   系统默认连续完成思考、生产、分发准备，只在关键节点拉人 review。

3. `Full Autopilot`
   系统持续自动产出、自动发布、自动复盘，用户主要处理异常。

### 4.2 推荐

推荐明确走：

> `Review-first Agent -> Managed Autopilot -> Full Autopilot`

原因很简单：

- 直接停在 Copilot，会继续把产品做成更聪明的工作流工具
- 直接跳到 Full Autopilot，又会因为评估、风控、分发不稳而失控
- 从 Review-first 演进，最符合“前期可干预、后期越来越少干预”的目标

---

## 5. 最终呈现方式

### 5.1 前台：Campaign Mission Control

默认首页不应该再问“你要用哪个工具”，而应该问：

> 你今天要推进哪个 campaign？

前台默认只展示五类信息：

- `新建 Campaign`
- `进行中的 Campaign`
- `需要你处理的少数事项`
- `系统正在做什么`
- `近期人工反馈与下一轮调整方向`

首页主视觉应该是：

> Campaign 列表 + Agent 状态 + 待决事项

而不是一排工具入口。

### 5.2 后台：Advanced Studio

所有专业能力继续保留，但退到第二层表面。

`Advanced Studio` 主要承接：

- Editor 精修
- 素材替换
- 深度 variant 微调
- prompt / script 微调
- 发布参数覆盖
- 手工重发 / 手工排期

原则是：

> 默认体验服务市场和运营，专业能力服务高级协作者。

---

## 6. 最短主链路

默认主链路应该被重写为：

`Brief -> System Plans -> System Produces -> System Distributes -> Human Feedback -> System Improves`

这里有几个重要变化：

1. 用户默认不再逐步手工选 hook、CTA、variant 数量
2. 系统要先决定“这轮最合理的素材组合”
3. 发布不再是额外步骤，而是主链路的一部分
4. 学习先基于人工反馈，不假装短期已经拥有完整效果监控

---

## 7. Campaign Brain 的定义

这个系统的大脑不应该被理解成一个会聊天的 Agent，而应该被理解成一个会做创意与分发决策的运营系统。

### 7.1 推荐形态：策略引擎脑

不走纯模板脑，也不直接走全自主脑，推荐走 `策略引擎脑`。

它要稳定回答四个问题：

- 这轮 campaign 应该测什么
- 该产哪些素材类型
- 每类产多少
- 哪些自动发，哪些需要人看一眼

### 7.2 四层结构

1. `Brief Interpreter`
   把 brief 变成标准化 campaign 输入。

2. `Creative Planner`
   决定要产什么素材组合、测什么角度、优先什么渠道。

3. `Publish Controller`
   决定哪些内容直接进入分发，哪些进入待决队列。

4. `Human Feedback Loop`
   记住人的判断，让下一轮更懂这个 campaign。

---

## 8. 核心对象模型

平台核心对象建议统一为 `Campaign`，其下只保留五类子对象：

### 8.1 Campaign Profile

用户真正输入的边界条件：

- goal
- product / service
- audience
- channels
- brand constraints
- compliance rules
- automation level

### 8.2 Campaign Plan

系统给出的行动计划：

- 这轮测哪些 angle
- 产哪些素材类型
- 每类产多少
- 哪些先小测
- 哪些渠道优先

### 8.3 Asset Packs

按目标生成的一组内容包，而不是孤立单条素材。

后续可以包含：

- 短视频
- 静态图
- 封面
- 标题 / caption
- 平台适配版本

### 8.4 Distribution Batches

分发层对象，记录：

- 发到哪
- 什么时候发
- 自动发还是待确认
- 当前状态
- 关联哪组素材

### 8.5 Feedback Records

短期不做重效果监控，先记住人工反馈：

- 这个方向是否继续
- 哪类 angle 更值得保留
- 哪类表达不符合品牌
- 哪类素材类型不适合这个 campaign
- 下轮更激进还是更保守

---

## 9. 状态机建议

Campaign 状态建议从一次性任务改成持续任务：

`Draft -> Thinking -> Producing -> Reviewing -> Publishing -> Feedbacking -> Iterating`

并补齐四个异常态：

- `Needs Input`
- `Paused`
- `Blocked`
- `Failed`

这样首页的“待决事项”才能真正成立。

---

## 10. 短期不做表现监控，先做人工反馈记忆

短期内做不了“哪些效果更好”的自动监控，就不要假装自己已经有完整 Learning Loop。

推荐把学习层短期降级为：

> `Human Feedback Loop`

### 10.1 先记住什么

- 这个 campaign 方向对不对
- 哪类 angle 更值得继续
- 哪类素材类型不适合
- 哪条内容可以直接发布
- 哪条内容需要修改
- 哪类表达不符合品牌或渠道
- 下轮更偏向哪种风格或目标

### 10.2 先做哪三层反馈记忆

1. `Campaign-level Feedback`
2. `Asset-type Preference`
3. `Guardrail Feedback`

### 10.3 页面呈现也要同步调整

短期默认不强调“哪条跑得最好”，而改成：

- `近期人工反馈`
- `系统已吸收的偏好`
- `下一轮准备如何调整`

---

## 11. 人工干预应该出现在哪里

默认体验不该让人驱动流程，而该让系统驱动流程，人只处理关键判断和例外。

### 11.1 必须有人介入的四类节点

1. `定义边界`
   brief、目标、限制、自动化等级。

2. `处理低置信或高风险决策`
   新 campaign、敏感 claim、新平台首发等。

3. `给出简短反馈`
   继续、换角度、更激进、更品牌、不要这种表达。

4. `处理异常`
   发布失败、渠道变化、质量不达标、结果偏离预期。

### 11.2 自动化等级

- `Assist`
  发布前都要确认。
- `Managed Autopilot`
  低风险自动继续，高风险进入待决队列。
- `Full Autopilot`
  默认自动生产与分发，人只处理异常和反馈。

---

## 12. 对现有能力的影响

当前已经落地的 `strategy / variant / editor handoff` 方向没有错，但定位要变：

- 它们不再是平台主舞台
- 它们成为 `Campaign Brain` 和 `Advanced Studio` 的底层能力
- 以后呈现时要以“系统决定与系统执行”为主，而不是以“用户微调每个细节”为主

也就是说，现有能力不推翻，但需要重新编排和降噪。

---

## 13. 下一阶段建议

推荐下一阶段先做的，不是继续扩更多专业调参，而是完成一次产品骨架重构：

1. 把首页和导航重写成 `Campaign Mission Control`
2. 把 `Campaign` 提升为唯一主对象
3. 把 `Campaign Brain` 的输入、计划、分发、反馈对象模型固定下来
4. 把专业细调能力整体下沉到 `Advanced Studio`
5. 把短期学习层定义成 `Human Feedback Loop`

---

## 14. 成功标志

如果这次方向重构成功，团队后续讨论的重点会明显变化：

- 不再主要讨论“哪个编辑功能更强”
- 不再主要讨论“用户要不要再多一个调参项”
- 而是开始讨论“系统该如何更稳定地产出更能跑的内容”

最终如果市场和运营用户能用更少点击完成更多 campaign 推进动作，并且逐步愿意把更多决策交给系统，就说明产品真正走上正轨了。
