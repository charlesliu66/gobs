# 设备台 · 批量养号 · 需求确认书

**版本**：与产品确认一致（2026-03-24）  
**关联能力**：在**设备台**增加「批量养号」；调度与执行**复用 GeeLark TikTok 养号自动化**（与现网 `/task/add` 养号任务、`taskAddWarmup` 能力一致）。

---

## 一、入口与主流程

| 项 | 说明 |
|----|------|
| 入口 | **设备台**增加「**批量养号**」，点击后打开**弹窗** |
| 弹窗内 | 选择设备 → 配置养号参数（浏览时长区间等）→ 选择 **立即开始养号** 或 **定时养号** → 确认 |

---

## 二、设备选择

- 仅面向**TikTok 养号**场景；支持**多选**、**全选**（全选范围：当前列表或当前筛选结果，实现时与设备页一致即可）。
- 设备状态：若产品需禁止选关机设备，可在 UI 中禁用；若需允许选中后自动处理，见下文「离线策略」。

---

## 三、养号行为（已确认）

| 项 | 结论 |
|----|------|
| 行为 | **随机浏览短视频**（与 GeeLark 养号任务能力对齐，见 `action` 等参数） |
| 浏览时长 | 用户设置**时长区间** `[T_min, T_max]`；每台设备在区间内**随机抽取一个实际浏览时长**，以模拟真人 |
| 底层实现 | **复用 GeeLark TikTok 养号自动化**（现有 `task/add` + `taskType: 2` 养号链路，与 `web/lib/geelark.ts` 中 `taskAddWarmup` 一致） |

---

## 四、执行方式

### 1. 立即开始养号（已确认文案）

- 按钮/选项文案：**「立即开始养号」**（不使用「即刻发布」等易混表述）。
- 确认后对所选设备下发养号任务（具体排队策略与现网任务系统一致）。

### 2. 定时养号（每日）

- 支持配置**多条定时规则**（多个定时任务）。
- **时区**：一律按**用户浏览器本地时区**解释与展示时间（含日期、区间起止）。
- **重复**：至少支持「**每天**」在设定时间窗口内触发。
- **开始时间区间**：`[t_start, t_end]` 为**同一天内的时间段**；每台设备在该区间内**随机一个时刻**作为**当次养号开始时间**（模拟真人）。
- **同一设备、同一天**：**允许**存在**多条**定时规则（例如午间一段、晚间一段），各规则独立执行。

### 3. 任务管理

- 对已创建的定时任务：支持**列表展示**与**取消**（取消后不再按该规则触发）。
- 取消时**是否中断当前正在执行的单次任务**：若本期未单独约定，建议与现网「取消任务」行为一致（可对接 GeeLark `task/cancel`）。

## 五、设备状态与任务前后（已确认）

| 场景 | 结论 |
|------|------|
| 设备离线 | **跳过**该次执行，**但必须在日志/界面中明确告知用户**（哪台设备、哪条规则、哪次计划被跳过及原因）。 |
| 执行前 | 若计划执行时设备为关机/离线：需**先启动设备**再下发/执行养号（与「设备不会离线」前提一致；若仍遇异常则走「跳过 + 日志」）。 |
| 执行后 | **每台设备在本次养号任务执行结束后关机**（与现有「关机」能力对齐，如设备台 `stop` 等）。 |

## 六、异常与可观测性（已确认）

| 场景 | 结论 |
|------|------|
| 因离线等原因跳过 | **跳过** + **日志/提示必须可见**，便于用户补操作或改规则 |

## 七、实现对齐（开发备注）

- **GeeLark**：养号任务创建继续走 **`task/add` + 养号任务类型**（当前代码中 `taskAddWarmup`）；**随机时长**在服务端或调度层根据用户给定区间生成**单次任务的 `duration`**（分钟等）再调用接口。
- **定时**：浏览器本地时区 → 转换为调度用的 **Unix 时间戳**（`scheduleAt`）时需与现有定时任务一致，避免夏令时与跨日边界错误。
- **关机**：任务完成回调或轮询任务状态后对该**台**设备触发 `phone/stop`（或项目内已有封装）；**每台**养号任务结束后**分别**关机。
