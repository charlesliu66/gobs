## 二、共享文件操作规则（Shared Files）

以下文件两边都可能需要修改，必须遵守操作规范：

### `h5-video-tool-api/src/index.ts`（路由注册入口）

**规则：只追加，不修改已有行。**

```typescript
// ✅ 正确：在文件末尾追加新路由，不动其他行
app.use('/api', editorFromProductionRouter);   // 制片 Agent 追加
app.use('/api', musicBeatRouter);              // 剪辑 Agent 追加

// ❌ 错误：修改已有路由注册行，或调整顺序
```

**约定**：每个新功能建**独立路由文件**，在 `index.ts` 只加一行 `use()`，不改现有内容。

---

### `h5-video-tool/src/api/editor.ts`（前端 API 层）

**规则：只在文件末尾追加新函数，不修改已有函数。**

```typescript
// ✅ 正确：末尾追加
export async function createEditorProjectFromProduction(...) { ... }  // 制片 Agent
export async function listExportFiles(...) { ... }                    // 剪辑 Agent

// ❌ 错误：修改已有的 startEditorExport / loadEditorProject 等函数
```

---

### `PRODUCT.md`（产品文档）

**规则：每次更新在 Changelog 顶部插入新版本块（`### vX.XX`），不修改其他版本的内容。**

## 三、新增文件命名约定

为避免命名冲突，新建文件必须带模块前缀：

| 模块 | 文件名前缀 | 示例 |
|------|-----------|------|
| 剪辑功能 | `editor` 或 `music` | `musicBeatAnalysis.ts` |
| 制片功能 | `production` 或 `screening` | `productionVersionService.ts` |
| 共用工具 | 需双方确认后命名 | — |

## 四、Git 分支与合入规则

### 分支命名

```bash
# 剪辑 Agent
git checkout -b feat/editor-ai-v2

# 制片 Agent  
git checkout -b feat/production-screening
```

### 每日同步（必须执行）

```bash
# 每天开始工作前，rebase 到最新 main
git fetch origin
git rebase origin/main

# 若有冲突，立即通知对方协商解决
```

### 合入顺序（降低风险）

```
第 1 个合入：制片 P0（放映室）
  原因：纯前端新增组件，零冲突风险

第 2 个合入：剪辑优化（方向1+5+3）
  原因：完全独立文件，不依赖制片改动

第 3 个合入：制片 P1（版本历史）
  原因：涉及后端存储改造，风险最高，留到最后

第 4 个合入：制片 P2（直通剪辑器）
  原因：新增跨模块接口，需双方联调后合入
```

### 合入前必须通过的检查

```bash
# 后端编译
cd h5-video-tool-api && npx tsc --noEmit

# 前端编译
cd h5-video-tool && npx tsc --noEmit

# 两个都通过才允许 merge，任何一个报错必须先修复
```

## 五、制片 Agent 绕开禁区的替代方案

制片 P1（版本历史）需要改数据结构，但 `studioPipeline.ts` 和 `productionTypes.ts` 是禁区。

**推荐绕路方案：**

```
❌ 禁止：直接修改 productionTypes.ts 里的 ProductionShot 接口

✅ 替代：
  1. 新建 src/types/productionShotVersions.ts
     定义 VideoVersion 接口 + ShotWithVersions 扩展类型

  2. 在 routes/studio.ts 中实现版本追加逻辑
     （在路由层处理文件重命名 + JSON patch，不碰 studioPipeline.ts）

  3. 存储层用文件命名约定区分版本：
     shot_<id>_v1_<timestamp>.mp4
     shot_<id>_v2_<timestamp>.mp4
```

## 六、冲突发生时的处理流程

```
1. 发现冲突 → 不要强行解决，先停下来
2. 查看冲突文件 → 确认是哪个模块的代码
3. 属于对方模块的改动 → 保留对方的，自己的重新适配
4. 共享文件冲突 → 两边各自的追加内容都要保留，手动合并
5. 合并后重新跑编译检查，通过才提交
```
