# 每日工作复盘（Daily Review）

## 触发条件

用户说以下任意一种：
- "日报" / "每日复盘" / "daily review"
- "帮我复盘今天的工作"
- "今天做了什么"
- "/daily-review"

## 执行流程

### Phase 1：数据采集（并行读取）

同时读取以下文件，不遗漏：

```
1. git log --oneline --since="今天00:00" --until="明天00:00"   ← 今日所有 commit
2. PRODUCT.md                                                   ← changelog 数据源
3. docs/ 下今天新建或修改的 .md 文件                              ← 方案文档上下文
4. .claude/memory/feedback.md                                   ← 现有行为规则
5. docs/daily-reports/ 下最近的日报                               ← 用于趋势对比
6. docs/TASK-INDEX.md                                           ← 任务完成状态
```

**采集要点**：
- git log 用 `--since` 和 `--until` 精确到当天
- 扫描 `docs/` 目录下今天修改的方案文档（用 `git diff --name-only HEAD~N` 或文件修改时间判断）
- 如果有上一份日报，记下其中的关键指标（Bug Fix 占比、协作评分等）用于对比

### Phase 2：分析生成

#### 2.1 功能分类统计

遍历今天 PRODUCT.md 中新增的 changelog 条目，按以下三类统计：

| 类别 | 判断标准 |
|------|---------|
| 新功能 / 体验优化 | 标题含 Feature / UX / feat |
| Bug Fix / 修复 | 标题含 Fix / Bug / 修复 |
| 架构重构 / 技术债 | 标题含 Refactor / Chore / arch |

计算各类占比。**Bug Fix 占比是核心指标**——目标是逐日降低。

#### 2.2 同类问题归因

扫描今天的 changelog，检查是否存在：
- 同一功能/模块被修改 2 次以上（如"即梦并发"改了多次）
- 同一 Bug 的不同症状分散在多个版本中
- 修复 A 引入了新的 Bug B

对每组同类问题，分析**根因**和**如何一次到位**。

#### 2.3 趋势对比（如果有上一份日报）

与上一份日报对比以下指标：

| 指标 | 目标趋势 |
|------|---------|
| Bug Fix 占比 | ↓ 下降（目标 < 20%） |
| 同类问题重复次数 | ↓ 归零 |
| 端到端验证评分 | ↑ 上升（目标 4★） |
| 版本数 | 适中（不追求多，追求稳） |

#### 2.4 新规则提炼

判断是否需要新增 feedback.md 规则：
- **触发条件**：同类问题在今天出现 ≥ 2 次，或某个 Bug 的根因具有普遍性
- **格式**：遵循 feedback.md 现有格式（规则 N / Why 反面教材 / How to apply）
- **不要滥加**：只有真正具有"下次还会犯"风险的教训才值得写入规则

#### 2.5 明日 TODO 提取

从以下三个来源提取明天的待办：

**来源 A — 未完成的方案文档**：
- 扫描 `docs/` 下的方案文档，找到标注了优先级但今天未实现的条目
- 检查 TASK-INDEX.md 中是否有未完成任务

**来源 B — 架构优化总清单剩余项**：
- 读取 `docs/架构优化总清单-*.md`，列出未完成的 P1/P2 项

**来源 C — 今日开发中发现的改进点**：
- 从 changelog 的 Root Cause 分析中提取"应该做但没做"的改进
- 从 Phase 2.2 的同类问题分析中提取预防性改进

按优先级排序给出建议执行顺序。

### Phase 3：写入文件

#### 3.1 日报文件

写入 `docs/daily-reports/YYYY-MM-DD.md`，固定八个章节：

```markdown
# 日报 · YYYY-MM-DD（周X）

> 协作模式：人 + AI Agent（Cursor / Claude Code）
> 统计口径：当日 git commit + PRODUCT.md changelog

## 一、今日产出概览
（表格：Git Commits / 版本迭代 / 新功能 / Bug Fix / 架构重构 / 新增文档）

## 二、完成功能清单
（按模块分组的版本列表，含版本号、内容、性质标签）

## 三、经验教训
（高频致命类 + 新沉淀的规则）

## 四、Changelog 效率分析
（版本分布 + 同类问题反复迭代分析 + 格式问题）

## 五、高效的方面
（今天做得好的 3-5 点）

## 六、后续优化建议
（从分析中提炼的改进方向）

## 七、协作效率评分
（6 维度评分表 + 与上一份日报的趋势对比）

## 八、明日 TODO 建议
（三个来源：未完成方案 + 架构剩余项 + 开发中发现的改进点）
```

#### 3.2 记忆文档更新

根据分析结果，**按需**更新以下文件（没有变化则不改）：

| 文件 | 更新条件 |
|------|---------|
| `.claude/memory/feedback.md` | Phase 2.4 提炼出了新规则 |
| `.claude/memory/project.md` | 技术栈变化、任务状态变化、新增重要基础设施 |
| `.claude/memory/MEMORY.md` | 新增了记忆文件或索引项变化 |
| `CLAUDE.md` | 新增环境变量、新增关键配置项、工作流变更 |

**更新原则**：
- feedback.md 只追加不删改已有规则
- project.md 只更新事实性状态，不加主观评价
- CLAUDE.md 只加对未来所有会话都有用的配置信息

#### 3.3 PRODUCT.md 健康检查

扫描 PRODUCT.md 是否存在以下问题，有则修复：
- 重复版本号（同一个 vX.Y 出现两次）
- Markdown 格式错误（如 `**---**`）
- 「最后更新」标注与实际最新版本不一致
- 功能模块总览与 changelog 新功能不同步

### Phase 4：输出摘要

所有文件写入完成后，向用户输出一段简洁的摘要：

```
✅ 日报已生成：docs/daily-reports/YYYY-MM-DD.md

📊 今日概览：
- N 个 commit / N 个版本（vX.Y → vX.Z）
- 新功能 X% / Bug Fix Y% / 架构 Z%
- [趋势] Bug Fix 占比：上次 A% → 本次 B%（↑/↓）

📝 更新了以下文件：
- feedback.md：新增规则 N（简述）
- project.md：更新了 XXX
- CLAUDE.md：新增环境变量 XXX

📋 明日建议 TOP 3：
1. [高优] XXX
2. [中优] XXX
3. [低优] XXX
```

## 注意事项

- **不要编造数据**：所有统计必须基于实际 git log 和 PRODUCT.md，不要凑数
- **不要过度规则化**：feedback.md 不是越多越好，只加真正有"复发风险"的教训
- **日报要简洁**：每个章节控制篇幅，避免变成流水账；changelog 效率分析重在归因，不是罗列
- **明日 TODO 要可执行**：每条待办要有明确的输出物和预估工时，不要写"继续优化 XXX"这种模糊项
- **趋势对比是核心价值**：没有上一份日报时跳过对比，有的话务必做，这是衡量协作是否在变好的关键
