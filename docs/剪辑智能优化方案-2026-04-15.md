# AI 剪辑智能优化方案

> 版本：v1.0 | 日期：2026-04-15
> 背景：当前剪辑系统已能识别画面主角（如「盗贼」），但行为识别准确率低、画面与背景音乐契合度差。
> 本文档从**资深剪辑师视角**出发，系统梳理 7 个优化方向，并给出优先级建议与落地路径。

---

## 当前系统现状

### 现有工作流（简版）

```
用户输入自然语言意图
       ↓
意图路由（剪辑 / 闲聊）
       ↓
素材分析（音频能量 + Gemini 视觉打分）
  - 输出：score(0-10)、roles、scene、activity
       ↓
LLM 生成时间轴（基于候选片段 + score 排序）
       ↓
后处理 / 导出
```

### 已确认的优势
- ✅ 主角/角色识别：能正确找到「盗贼」「刺客」等职业镜头

### 已确认的问题
- ❌ **行为识别粗糙**：`activity` 只有单一标签（奔跑/打架/撤退），无强度、转折点等维度，LLM 无法区分「普通跑步」和「逃命奔跑」
- ❌ **叙事结构缺失**：系统以"高光分"堆砌片段，没有钩子→铺垫→高潮→收尾的节奏感
- ❌ **音画不同步**：BGM 是在视频片段选定后才叠加，卡点完全靠随机对齐，无节拍驱动
- ❌ **内容同质化**：连续输出同类行为（如 5 个打架片段），节奏单调

---

## 七个优化方向

---

### 方向一：行为细化分类（Behavior Taxonomy 二级体系）

**当前问题**：`activity` 是单一字符串（`打架` / `撤退`），LLM 无法感知强度与转折。

**优化方案**：将行为标注扩展为二级结构：

```typescript
// 当前：string
activity: "打架"

// 优化后：结构化对象
activity: {
  primary: "打架",          // 一级：搜索 / 打架 / 交互 / 撤退
  secondary: "击杀瞬间",    // 二级：补刀 / 爆头 / 连招 / 击退 / 团战 ...
  intensity: "high",        // 强度：low / mid / high
  isTurningPoint: true,     // 是否为叙事转折点（反杀、被围困、胜利时刻）
}
```

**二级行为词典（游戏场景示例）**：

| 一级 | 二级示例 |
|------|---------|
| 搜索 | 开箱、探索、拾取、侦查 |
| 打架 | 击杀瞬间、连招、团战、反杀、boss战 |
| 交互 | 开门、使用技能、与NPC对话 |
| 撤退 | 逃跑奔跑、缩圈追赶、安全撤离 |

**改动文件**：
- `frameVisionRank.ts`：在 prompt 中加入二级行为词典，要求 Gemini 输出结构化 activity
- `gameTaxonomy.ts`：扩展 activities 字段为嵌套结构
- `editorHighlightCandidates.ts`：基于 `isTurningPoint=true` 给候选窗口加权

**预估工作量**：2-3 天  
**优先级**：⭐⭐⭐⭐⭐（直接解决用户痛点）

---

### 方向二：两阶段叙事结构排片

**当前问题**：LLM 一次性选片+排片，没有叙事蓝图，片段堆砌感强。

**优化方案**：拆分为两个阶段

```
阶段 1 - "素材清单生成"
  对所有候选片段打上行为标签 + 时间戳
  输出：结构化素材清单（内容地图）

阶段 2 - "叙事编排"  
  基于素材清单，按叙事模板选片组合
  模板示例：钩子(5s) → 铺垫(15s) → 高潮(20s) → 收尾(10s)
```

**叙事模板库（可扩展）**：

| 模板名 | 适用场景 | 结构 |
|--------|---------|------|
| 经典高光 | 混剪爽点 | 钩子→高强度战斗→击杀高潮→结局 |
| 角色故事 | 角色宣传片 | 角色登场→能力展示→危机时刻→逆转 |
| 紧张剧情 | 生存/逃跑 | 安静→发现危机→逃跑→结果 |
| 节奏混剪 | 卡BGM | 完全由音乐节拍驱动，内容服从节奏 |

**改动文件**：
- `editorAgentService.ts`：增加 `buildContentManifest()` 函数（阶段1）
- `editorAgentService.ts`：修改 LLM prompt，注入素材清单 + 叙事模板（阶段2）

**预估工作量**：3-4 天  
**优先级**：⭐⭐⭐⭐

---

### 方向三：音乐先行 + 卡点剪辑（最关键）

**当前问题**：先选片再加BGM，节拍对齐纯靠运气。

**优化方案：音乐先行流程**

```
流程 A（当前）：素材分析 → 片段选择 → 叠加BGM
流程 B（新）：  生成/选择BGM → 分析节拍结构 → 基于节拍选片 → 卡点剪辑
```

**节拍分析服务（新建 `musicBeatAnalysis.ts`）**：

输入：BGM 音频文件路径  
输出：
```json
{
  "bpm": 128,
  "beatTimings": [0.0, 0.47, 0.94, 1.41, ...],
  "sections": [
    { "name": "intro",  "startSec": 0,   "endSec": 8,   "energy": "low" },
    { "name": "build",  "startSec": 8,   "endSec": 24,  "energy": "mid" },
    { "name": "drop",   "startSec": 24,  "endSec": 48,  "energy": "high" },
    { "name": "outro",  "startSec": 48,  "endSec": 60,  "energy": "low" }
  ]
}
```

**音乐段落 → 内容情绪映射规则**：

| BGM 段落 | 对应内容类型 | 建议镜头时长 |
|---------|------------|------------|
| intro (low energy) | 角色登场、场景建立 | 3-5s |
| build (mid energy) | 奔跑、探索、接近目标 | 2-3s |
| drop (high energy) | 战斗高光、击杀瞬间 | 0.5-2s（快切） |
| outro (low energy) | 结局、庆祝、收尾 | 3-5s |

**节拍对齐剪辑逻辑**：
- 在「drop」段：每个节拍点（约0.47s间隔）切一次，选取 `score ≥ 7` 且 `activity.primary = 打架` 的帧
- 在「build」段：每 2-3 拍切一次，选取 `intensity = mid` 的行为片段
- 在「intro」段：不卡节拍，优先选角色出场镜头

**技术实现**：
- 使用 Python `librosa` 库做 BPM 检测和节拍追踪（已有 `PYTHON_EXE` 环境变量）
- 后端新建 `musicBeatAnalysis.ts` 调用 Python 脚本

**改动文件**：
- 新建 `h5-video-tool-api/src/services/musicBeatAnalysis.ts`
- 新建 `h5-video-tool-api/scripts/beat_analysis.py`
- `editorAgentService.ts`：支持「先生成音乐 → 分析节拍 → 按节拍排片」模式
- 前端：`AgentPanel` 新增「🎵 音乐先行」模式开关

**预估工作量**：4-5 天  
**优先级**：⭐⭐⭐⭐⭐（用户明确需求）

### 方向四：切点质量优化

**当前问题**：片段的入点/出点是「分数最高帧的前后」，没考虑画面运动状态，切点突兀。

**优化方案**：在 `VisionFrameScore` 中增加切点维度：

```typescript
export interface VisionFrameScore {
  tSec: number;
  score: number;
  // 新增：
  motionBlur: boolean;      // 当前帧是否处于运动模糊（适合作为切点）
  cameraMotion: 'static' | 'pan' | 'zoom' | 'shake';  // 镜头运动类型
  isActionPeak: boolean;    // 是否为动作顶点（如出拳到位的瞬间）
  // 已有：
  roles?: string[];
  scene?: string;
  activity?: string;
}
```

**切点设计规则**：
- **动接动**：运动镜头切运动镜头（pan→pan），视觉流畅
- **动作顶点切**：在 `isActionPeak=true` 帧切入（如击杀瞬间），爆发感强
- **避免运动模糊帧作入点**：`motionBlur=true` 的帧适合作切点，但不适合作片段开头
- **镜头运动匹配**：连续镜头的 `cameraMotion` 应有一定多样性

**预估工作量**：2-3 天  
**优先级**：⭐⭐⭐

### 方向五：内容多样性约束

**当前问题**：LLM 倾向于堆砌同类高分片段，导致连续 5 个打架画面、节奏单调。

**优化方案**：在 LLM prompt 中加入硬性多样性规则：

```
多样性约束（LLM 必须遵守）：
1. 同一 primary activity 连续出现不超过 3 次
2. 每隔 3 个战斗片段，穿插 1 个非战斗片段（跑动、场景）
3. 每 10 秒内，不能有超过 3 次「快切」（单片段 < 1s）
4. 开头 5 秒内，必须有一个角色登场或钩子镜头
5. 结尾最后 3 秒，优先选择 score 最高 且 isActionPeak=true 的片段
```

**额外多样性维度（参考）**：
- 景别多样性：特写/中景/远景各保证一定比例
- 时间流动性：早/中/晚游戏进度各阶段都有覆盖

**改动文件**：
- `editorAgentService.ts`：`buildMainLlmPrompt()` 中注入多样性约束块

**预估工作量**：1 天  
**优先级**：⭐⭐⭐⭐

### 方向六：画面-音乐情绪对齐

**当前问题**：视频的视觉张力与BGM能量段落没有主动匹配，高潮音乐段可能对应平淡画面。

**优化方案**：在 `VisionFrameScore` 中加入情绪张力维度：

```typescript
tension: number;  // 0-10，与 score 分离：score=剪辑价值，tension=情绪强度
emotionTag: 'calm' | 'tense' | 'triumphant' | 'sad' | 'exciting';
```

**情绪张力标注规则（Gemini prompt 新增）**：
- `tension >= 8`：爆炸、死亡、逆转、高声对话
- `tension 4-7`：奔跑、追逐、准备战斗
- `tension <= 3`：静态场景、UI界面、对话

**排片时的情绪对齐逻辑**：
```
BGM 段落 energy=high  →  优先选 tension >= 7 的片段
BGM 段落 energy=mid   →  优先选 tension 4-6 的片段  
BGM 段落 energy=low   →  优先选 tension <= 4 的片段
```

**改动文件**：
- `frameVisionRank.ts`：扩展 prompt 增加 tension/emotionTag 评分
- `editorAgentService.ts`：排片时传入 BGM 段落结构，做情绪对齐

**预估工作量**：2-3 天（依赖方向三完成）  
**优先级**：⭐⭐⭐（配合方向三效果最佳）

---

### 方向七：用户反馈学习

**当前问题**：系统不知道用户喜好，每次从零开始，无法持续改善。

**优化方案**：轻量级偏好画像（不需要机器学习，纯规则积累）：

**收集数据点**：
- 用户保留/删除了哪些 activity 类型的片段
- 用户替换了哪段 BGM（说明自动选的不合适）
- 用户调整了哪些片段的时长（偏好长镜头 or 快切）
- 用户最终导出的视频结构（哪类片段占比最高）

**偏好画像结构**：
```json
{
  "username": "admin",
  "preferredActivities": ["击杀瞬间", "团战", "逃跑"],
  "avoidedActivities": ["开箱", "静态探索"],
  "avgClipDurationSec": 2.1,
  "preferFastCut": true,
  "preferredMusicStyle": "高能电子"
}
```

**使用方式**：
- 下次调用 LLM 排片时，注入偏好画像作为额外上下文
- 示例 prompt 追加：`"用户偏好：快节奏切换，偏好战斗击杀片段，平均 2 秒一切"`

**改动文件**：
- 新建 `userPreferenceService.ts`
- `editorAgentService.ts`：在构建 LLM prompt 时注入偏好数据
- 前端：在导出时静默上传行为统计数据（可选，需用户同意）

**预估工作量**：3-5 天（基础版 1 天）  
**优先级**：⭐⭐（长期收益，近期可以先做基础版）

## 优先级汇总与落地路径

### 优先级矩阵

| 方向 | 解决的核心问题 | 难度 | 优先级 | 建议阶段 |
|------|-------------|------|--------|---------|
| 方向 3：音乐先行+卡点 | 画面与BGM脱节 | 高 | P0 | 第一批 |
| 方向 1：行为细化分类 | 行为识别粗糙 | 中 | P0 | 第一批 |
| 方向 5：内容多样性约束 | 内容同质化 | 低 | P1 | 第一批 |
| 方向 2：两阶段叙事结构 | 片段堆砌无叙事感 | 中 | P1 | 第二批 |
| 方向 4：切点质量优化 | 切点生硬 | 中 | P2 | 第二批 |
| 方向 6：情绪对齐 | 视觉张力与音乐不匹配 | 中 | P2 | 第三批 |
| 方向 7：用户反馈学习 | 无个性化优化 | 高 | P3 | 第三批 |

### 推荐落地节奏

```
第一批（1-2 周）：
  ① 方向 1 - 行为细化：修改 frameVisionRank.ts prompt（1天）
  ② 方向 5 - 多样性约束：修改 LLM prompt 加规则（1天）
  ③ 方向 3 - 音乐先行基础版：Python beat 检测脚本 + 节拍注入排片（3天）

第二批（3-4 周）：
  ④ 方向 2 - 两阶段叙事（依赖方向1完成后更有效）
  ⑤ 方向 4 - 切点优化

第三批（5-6 周）：
  ⑥ 方向 6 - 情绪对齐（依赖方向3的BGM分析）
  ⑦ 方向 7 - 用户反馈学习
```

## 技术注意事项

### Python 依赖（方向三需要）
```bash
pip install librosa soundfile numpy
# librosa 包含 BPM 检测、节拍追踪、能量段分析
```

### 改动边界
- **禁止直接修改**的文件（见 CLAUDE.md）：`dreaminaVideo.ts`、`klingVideo.ts`、`veoPython.ts`、`studioPipeline.ts`
- 方向 1-5 的改动都在允许范围内（`frameVisionRank.ts`、`editorAgentService.ts`、`gameTaxonomy.ts`）

### LLM Token 消耗影响
- 方向 1（行为细化）：prompt 增加约 200 tokens/次，影响可忽略
- 方向 2（两阶段）：增加一次 LLM 调用（阶段1），token 消耗 x1.5
- 方向 3（音乐分析）：Python 本地处理，不消耗 LLM token
