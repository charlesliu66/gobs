# GOBS 平台中英文切换设计方案 v2

> 目标：让以英文为主的本地同事可以顺畅完成 GOBS 的核心流程，同时不破坏当前中文主流程、既有数据和高风险模型协议。
> 作者：Codex · 2026-04-21
> 状态：v2 评审稿
> 说明：本稿基于 `docs/i18n-中英文切换设计方案.md` 重写，重点收紧可实施性、项目约束兼容性，以及本地英文同事的真实使用体验。

---

## 1. v2 相比原方案的核心变化

v2 保留原方案的主方向，但做了 6 个关键收口：

1. **先收口协议，再铺翻译**
   - 先把 locale、header、错误码、格式化、引用 token 这些“基础协议”统一，再扩页面翻译。

2. **避免一阶段直接碰项目禁区**
   - 原方案把 `studioPipeline.ts`、`productionAssets.ts`、`productionTypes.ts` 作为主要改动点，但这些文件当前属于项目禁区。
   - v2 改为：Phase 1-2 先做前端壳层、客户端适配层、非禁区服务；需要动禁区文件的内容拆成后续单独 run。

3. **把“显示语言”和“模型协议”分开**
   - 英文用户看到的提示、按钮、引用说明可以是英文。
   - 真正发给即梦/可灵的 token 不要求一开始就改成英文；必要时保留中文协议 token，由适配层转换。

4. **把“服务端本地化”改成“错误码优先，本地化消息兜底”**
   - 与其大面积让后端直接返回不同语言的自由文本，不如优先统一 `errorCode`，前端按 locale 映射消息，服务端消息只做兜底。

5. **把“英文同事可用”作为单独目标，不等同于“全量英文翻译”**
   - 真正影响 local 同事的是登录、导航、主流程、空状态、错误提示、时间格式、引用输入方式，不是一次性翻完整个系统。

6. **加入“本地同事视角”的额外优化项**
   - 包括登录页可切语言、术语双语过渡、引用插入 UI、未翻译区域提示、英文帮助入口等。

## 2. 目标、范围与非目标

### 2.1 目标

- 支持 **中文 UI** 与 **英文 UI** 两种界面语言。
- 支持 **中文内容** 与 **英文内容** 两种 AI 输出语言。
- 保证中文老用户行为不回退。
- 保证英文同事可以完成至少以下 P0 流程：
  - 登录
  - 进入首页和导航
  - QuickFilm
  - ProductionWizard 主流程的基础理解
  - 基础错误识别
  - 基础分发文案生成

### 2.2 非目标

- 不在首阶段追求全仓库全量翻译。
- 不在首阶段修改禁区文件中的核心生产逻辑。
- 不在首阶段迁移所有数据库中文字面量。
- 不在首阶段要求所有模型都直接接受英文协议 token。

## 3. 约束与设计前提

### 3.1 项目约束

当前项目明确禁止直接修改下列核心文件：

- `h5-video-tool-api/src/services/studioPipeline.ts`
- `h5-video-tool-api/src/types/productionTypes.ts`
- `h5-video-tool-api/src/config/productionAssets.ts`
- 以及底层 Dreamina / Kling / VEO 服务文件

因此，v2 要求：

- **Phase 1-2 不依赖改动这些文件才能成立**
- 所有必须触碰禁区文件的需求，单独列成后续 run

### 3.2 现状前提

代码中已经存在部分语言能力和可复用基础：

- 前端已有集中式 `apiGet/apiPost/apiPatch/...` 封装，可统一带 locale header
- `TabDistribute` 已经有内容语言切换能力
- `sj-ui` 已有基于 locale 生成语言约束 prompt 的实现
- 若干页面仍存在大量 `toLocaleString('zh-CN')` 和硬编码中文文案

这意味着首阶段应该优先做：

- locale 规范化
- 统一格式化工具
- 客户端 header 注入
- 核心页面壳层翻译

## 4. v2 总体架构

### 4.1 双维度语言模型

- `uiLocale`
  - 控制导航、按钮、表单标签、Toast、空状态、错误文案、时间格式等界面文本
  - 当前支持：`zh-CN`、`en`

- `contentLocale`
  - 控制 AI 产出的剧本、prompt、caption 等内容语言
  - 当前平台级只定义：`zh`、`en`
  - 个别模块可扩展自己的输出语言枚举，例如分发页的 `EN/CN/TH/ID`

### 4.2 规范化策略

统一做 locale normalize，避免 `en`、`en-US`、`zh`、`zh-CN` 混乱传播：

```ts
type UiLocale = 'zh-CN' | 'en';
type ContentLocale = 'zh' | 'en';

function normalizeUiLocale(raw?: string | null): UiLocale {
  const v = String(raw || '').toLowerCase();
  if (v.startsWith('en')) return 'en';
  return 'zh-CN';
}

function normalizeContentLocale(raw?: string | null): ContentLocale {
  const v = String(raw || '').toLowerCase();
  if (v.startsWith('en')) return 'en';
  return 'zh';
}
```

### 4.3 默认值策略

首次进入时：

1. 先读 `localStorage`
2. 没有则读浏览器 `navigator.language`
3. 再兜底 `zh-CN`

默认关系：

- `contentLocale` 默认跟随 `uiLocale`
- 只有用户明确改过内容语言时，才分离保存

建议存储：

```ts
localStorage['gobs_ui_locale'] = 'en'
localStorage['gobs_content_locale'] = 'zh'
```

### 4.4 Header 协议

v2 统一如下约定：

- `X-UI-Locale: zh-CN | en`
- `X-Content-Locale: zh | en`

`Accept-Language` 仅用于首次推断默认值，不作为 AI 生成语言的权威来源。

### 4.5 错误处理协议

优先返回：

```json
{
  "success": false,
  "errorCode": "MISSING_REQUIRED_FIELD",
  "error": "请提供 prompt"
}
```

前端处理顺序：

1. 有 `errorCode`：按 `uiLocale` 本地翻译
2. 无 `errorCode`：展示服务端 `error`
3. 网络异常：走统一网络错误映射

这样比“后端所有路由都直接国际化自由文本”更稳，也更适合渐进迁移。

### 4.6 时间与数字格式策略

禁止再在业务组件里直接写：

- `toLocaleString('zh-CN')`
- `toLocaleDateString('zh-CN')`
- `toLocaleTimeString('zh-CN')`

统一改为工具层：

```ts
formatDateTime(date, uiLocale)
formatDate(date, uiLocale)
formatNumber(num, uiLocale)
```

这样能避免：

- 散落替换不全
- 同一页面格式不一致
- 后续新增页面继续写回硬编码

## 5. 显示文本与模型协议分层

这是 v2 最重要的新增点。

### 5.1 不要把模型协议直接暴露给英文用户

原方案默认让英文用户直接输入：

- `@image1`
- `[Image1]`

然后后端再 back-translate 成中文协议 token。

这条路可以做，但对本地同事仍然不够友好，因为用户仍然要理解“模型引用协议”。

### 5.2 v2 建议的更优方案

把“引用关系”视为结构化能力，而不是让用户手输协议 token。

UI 层推荐：

- 素材卡上直接展示英文标签：`Image 1`, `Video 1`, `Audio 1`
- 点击即可插入
- 或者在输入框内用 mention/chip 形式选择引用
- 最终提交前由 provider adapter 生成各模型需要的 token

抽象层示意：

```ts
type RefMention = {
  id: string;
  kind: 'image' | 'video' | 'audio';
  index: number;
};
```

Provider adapter 负责：

- Dreamina: `@图片1`
- Kling Omni: `【图片1】`
- 内部英文展示: `Image 1`

### 5.3 渐进实施建议

考虑当前代码现实，v2 推荐分两步：

- **Phase 1**
  - 用户界面先保持现有引用协议
  - 但提示文案、按钮、素材标签、帮助说明改成双语或英文友好
  - 提供“点击插入引用”能力，尽量减少手打 token

- **Phase 2**
  - 再引入统一 `RefMention` 抽象
  - 提交前按模型 provider 做 token render

这样比“一次性全面切到 `@imageN`”更低风险。

## 6. 页面与能力分期

### Phase 0：协议与基础设施

目标：先把“语言基础设施”搭起来，不追求大面积翻页。

内容：

- 引入 `react-i18next`
- 建立 `locales/zh-CN` 与 `locales/en`
- 增加 `i18n.ts`
- 建立 locale normalize 工具
- 建立时间/数字格式化工具
- 在 `api/client.ts` 统一注入 `X-UI-Locale` 与 `X-Content-Locale`
- 增加前端错误码映射层

通过标准：

- 切换语言后无需刷新
- 请求能稳定带 locale headers
- 至少 3 个核心组件能使用统一格式化工具

### Phase 1：壳层与首屏可用

目标：让英文同事至少能“进得去、看得懂、知道点哪里”。

优先页面：

- 登录页
- Layout / 导航
- 首页
- 通用 Toast / 弹窗按钮 / 空状态

重点：

- 登录页必须支持语言切换，不能只在登录后切
- 导航文案、页面标题、常见按钮先翻
- 常见错误文案先翻

### Phase 2：P0 创作路径

目标：让英文同事能走通最关键创作流程。

优先页面：

- `QuickFilm.tsx`
- `ProductionWizard.tsx`
- 核心 `Step*.tsx`
- `TabDistribute.tsx` 的文案和状态

重点：

- UI 文案国际化
- `contentLocale` 在核心内容生成处打通
- 保持中文流程不回退

说明：

- 若某段内容生成依赖禁区文件，先通过外围参数、前端提示、局部适配做最小可用，不在这一阶段强行动核心服务

### Phase 3：解析器与适配层增强

目标：提升英文内容链路的稳定性。

内容：

- `storyboardParser.ts` 从“中文正则”升级为“时间码优先的语言无关解析”
- 多模态引用适配层落地
- provider-specific prompt token render
- 分镜 / 参考图 / caption 的内容语言一致性检查

### Phase 4：扩展页面与后端规范化

内容：

- History / Gallery / Asset Library / Editor
- 后端更多路由补 `errorCode`
- 统一错误码表
- 逐步减少路由里的中文自由文本

### Phase 5：服务端偏好与更完整的长期方案

内容：

- 用户偏好持久化接口
- 更多页面翻译
- 文档中心 / 使用帮助英文版
- 如需，再推进数据库枚举 key 英文化和迁移

## 7. 需要调整的原方案点

### 7.1 用户偏好同步到后端

原方案里提到：

- `PATCH /api/auth/preferences`

但当前项目里没有这个接口，也没有清晰的用户偏好持久化模型。

v2 建议：

- 首阶段只做 `localStorage`
- 等协议、前端体验稳定后，再单独增加服务端偏好同步

### 7.2 API 国际化方式

原方案倾向于让后端直接按 header 返回不同语言的错误文本。

v2 建议：

- **错误码优先**
- **前端翻译优先**
- 服务端原始错误文本只兜底

原因：

- 更适合当前已存在的大量中文路由
- 更适合渐进式迁移
- 更利于日志、排障和回归测试

### 7.3 `未分类` 的数据库默认值

原方案建议尽快改为 `uncategorized`。

v2 建议：

- 首阶段不改数据库默认值
- 先在展示层做映射
- 等有 migration 计划时再统一改存储

原因：

- 当前打标服务仍直接写 `'未分类'`
- 提前改存储语义，会引入历史兼容和查询口径问题

### 7.4 `@imageN` 直接对用户开放

原方案的想法合理，但对英文用户来说仍然太“工程化”。

v2 建议：

- 短期：保留协议兼容，优化提示与点击插入
- 中期：抽象成结构化引用
- 长期：彻底把 provider token 从用户输入层剥离

## 8. 本地英文同事视角：还值得继续优化的点

站在本地同事视角，除了“能切英文”，还有以下点非常值得做。

### 8.1 登录前就能切语言

如果语言切换只放在 `Layout`，那英文同事连登录页都看不懂。

建议：

- 登录页右上角直接放语言切换
- 登录失败提示、占位符、按钮文案同步翻译

这是英文同事真实体验里的 P0。

### 8.2 术语要做双语过渡，而不是生硬直译

像这些词：

- 一键成片
- 高级制片
- 风控大师
- 成片
- 分镜

如果直接翻成英文，local 同事未必能立刻理解它在产品里的真实含义。

建议：

- 前 1-2 个版本采用“双语过渡”或 hover glossary
- 例如：
  - `Production Wizard（高级制片）`
  - `Storyboard（分镜）`
  - `Final Video（成片）`

### 8.3 不要要求用户理解模型协议

英文同事通常不想知道：

- 即梦为什么要 `@图片1`
- 可灵为什么要 `【图片1】`

他们只想知道“这里引用第一张图”。

建议：

- 用 chip / picker / click-to-insert 取代手写 token
- 至少也要把帮助文案写成英文说明，而不是直接暴露中文协议

### 8.4 要允许“English UI + Chinese Content”一键切换

这不是边缘需求，而是高频需求。

典型场景：

- 英文同事操作系统
- 面向中国市场生成视频
- 仍希望 prompt / 脚本是中文

建议：

- 在入口处提供清晰预设：
  - `English UI + Chinese Content`
  - `English UI + English Content`
  - `中文界面 + 中文内容`

比让用户自己理解两个独立开关更友好。

### 8.5 未翻译区域要有明确提示

英文同事最怕的不是“有少量中文”，而是“不知道这个中文是不是 bug”。

建议：

- 未完成国际化的区域，用轻量标记说明：
  - `CN only for now`
  - `Translation in progress`

这样比静默混入中文更可控。

### 8.6 帮助说明要更任务化

本地同事不一定懂中国视频生产语境。

建议：

- 在核心流程旁增加简短英文解释：
  - 这一步会产出什么
  - 下一步用来做什么
  - 输入应该写什么粒度

尤其是：

- QuickFilm
- ProductionWizard
- 分镜与引用素材
- 发布文案生成

### 8.7 时间、状态和结果信息要更“可扫描”

英文同事对中文状态流转尤其敏感。

建议：

- 所有任务状态统一使用 locale-aware label
- 时间统一格式
- 长流程增加更明确的 stage label

例如：

- `Analyzing story`
- `Generating character setup`
- `Rendering storyboard`

而不是只替换按钮文案。

### 8.8 最好有一份英文快速上手

就算 UI 翻了，产品术语和工作流仍有门槛。

建议：

- 提供一页英文 onboarding
- 包括：
  - 核心模块说明
  - 推荐路径
  - 常见失败原因
  - 多模态引用怎么用

## 9. 风险矩阵

| 风险 | 等级 | 说明 | v2 缓解方式 |
|---|---|---|---|
| 直接碰禁区核心文件 | 高 | 与项目约束冲突，风险不可控 | 首阶段改壳层和适配层，禁区需求拆单独 run |
| locale 协议混乱 | 高 | `en`/`zh-CN`/`Accept-Language`/自定义 header 混用 | 明确 `X-UI-Locale` 与 `X-Content-Locale` |
| 即梦/可灵协议 token 不兼容 | 高 | 模型可能只认中文 token | UI 与协议分层，provider adapter 转换 |
| 英文同事仍需手写协议 token | 中 | 会造成体验和学习成本问题 | click-to-insert，后续 mention/chip 抽象 |
| 只翻页面，不翻状态和错误 | 中 | 用户依然难以完成任务 | 错误码映射 + 状态文案先行 |
| 历史中文数据与新英文数据混用 | 中 | 显示体验不一致 | 不翻旧数据，增加 language/source 标注 |
| 数据库存储值过早迁移 | 中 | 带来兼容和回归成本 | 首阶段只做展示映射 |

## 10. 验证矩阵

### 10.1 功能验证

- 英文登录页可正常使用
- 登录后语言立即生效，无需刷新
- `uiLocale=en + contentLocale=zh` 时：
  - UI 为英文
  - 内容生成默认仍走中文
- `uiLocale=en + contentLocale=en` 时：
  - UI 为英文
  - 支持英文内容链路
- 中文老用户路径不变

### 10.2 协议验证

- 所有 API 请求都带上 locale headers
- 无 header 时默认值稳定
- 错误码存在时前端能正确翻译
- 无错误码时仍可展示服务端原文

### 10.3 体验验证

- 英文同事首次登录无需解释即可找到主流程入口
- 英文同事能理解“内容语言”和“界面语言”的区别
- 不需要手动记忆 `@图片1` 才能完成基础引用

### 10.4 回归验证

- 中文路径不回退
- 时间格式不再出现硬编码 `zh-CN`
- 未翻译页面有明确状态标识，不误导用户

---

## 11. 建议的落地顺序

如果只看投入产出比，建议顺序如下：

1. `i18n.ts` + locale normalize + header 注入 + format helper
2. 登录页 + Layout + Toast + 常见错误
3. QuickFilm + ProductionWizard 的壳层文案
4. `TabDistribute` 与内容语言联动梳理
5. 引用 token 的点击插入能力
6. 分镜解析器语言无关化
7. 后端错误码体系
8. 服务端偏好持久化

---

## 12. 结论

v2 的核心判断是：

- **这不是一个“先全量翻译，再慢慢修协议”的问题**
- **而是一个“先把语言协议、错误协议、引用协议和用户体验接口收口，再逐页扩展翻译”的问题**

从本地英文同事的角度，最值得优先优化的并不是“180 个文件尽快翻完”，而是：

- 登录前就能切语言
- 主流程术语能看懂
- 错误信息能理解
- 不需要手打中文模型协议
- 可以方便地使用 `English UI + Chinese Content`

如果这几件事先做好，系统即使还有部分中文残留，本地同事也已经能稳定使用；反过来，如果只是翻更多页面，但没有解决这些关键交互，体验提升会很有限。
